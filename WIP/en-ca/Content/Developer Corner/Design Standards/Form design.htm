Form design standards

Overview

These guidelines are provided to help design forms that provide a quality user experience, optimize the use of screen real estate, and minimize the number of clicks.

GCPM form headers and footers are standardized, and typically should not be modified. This provides consistent information to users. The data that varies from one record to the next is inserted into the body section.

Note: labels are displayed above the field data.
Form Header

The header is located at the top of the form and include these standard elements:

  1. Unique ID (if applicable)
  2. Owning (Parent) entity – entity reference
  3. Owner (person responsible)
  4. Status reason

Note: information forms (such as the form to display Eligibility Criterion | PJ), do not have headers.

Form Footer

The footer is located at the bottom of the form, and include these standard elements:

  1. Created By
  2. Created On
  3. Modified By
  4. Modified On

Quick view (summary of the parent entity)

Location: Top of the body

Purpose: Provides read-only context from the parent record

Form Body

As you design a form and the control layout, consider how the user will interact with the form. and the effects scrolling, field tabs and other navigation features could have on the experience.

Each tab can have a different number of columns. Use a single-column layout for your form when:

  • You want to encourage users to fill out the form in a specific order.
  • The form spans multiple pages.
  • The app displays additional notes and info, instructions, or branding and advertising

See also: form layout from Microsoft MSDN Guidelines for form layouts*)

Field labels

Do not truncate field labels!

Most controls need a label.

  • Place labels above controls or to the left of a control.
  • Use the same style of label placement for all controls in a form.
  • In general, place labels above controls. When using multi-column form layouts, always place labels above controls. When placing labels above controls, it:
  • helps establish the relationship between a label and its control
  • helps create a clean layout, as all labels and controls can be left-aligned.
  • makes it easier to present long strings and handle localization and accessibility issues.
  • Placing labels to the left of controls when:
  • In single-column forms where vertical real estate must be conserved
  • The label strings are short and approximately the same length.
  • There is enough horizontal space for the longest label string (in any locale).
  • Placing labels inline (the text appears in the field, in grey font) when:
  • The form pattern is commonly understood by a wide audience (e.g., a user login or a password entry field).
  • The label can easily be implied or construed after the data has been entered in the input field (since hint text disappears after data is entered).
  • There are a limited number of input fields.
