Server side development guide

Please note that any of the following can be changed, hopefully with proper notice. If you are unsure about any of the guidelines below, please ask your team lead for more information.

Getting Started

Please download and use our EGCS CRM templates as a starting point for all your development work. This will ensure that the project is adhering to the guidelines set out below.

EGCS Development

Please download the EGCS CRM templates on WD's servers and start your development from the templates installed.

Currently available templates:
  • Solution Templates EGCS.CRM.Framework – load this solution first if you don't have the shared library on your machine.
  • Project Templates
    • EGCS.CRM.Plugins.SimplePlugin – this sets up a simple plugin project based on our custom plugin base, plugin context, and logging capabilities.
    • EGCS.CRM.Plugins.AutoPlugin– this sets up the same project as the simple plugin but includes the auto plugin step registration feature installed. You will have to configure the auto plugin step registration for it to know how to create your plugin steps.
    • EGCS.CRM.Workflows.SimpleWorkflow – this sets up a simple custom workflow activity project based on our own custom workflow base and workflow context.
  • Item Templates
    • EGCS.CRM.Items.Constant – this item template adheres to our custom Constant class standard. This will prefix your file and class name with Const.
    • EGCS.CRM.Items.Enumeration – this item template adheres to our custom Enumeration class standard. This will prefix your file and class name with Enum.
    • EGCS.CRM.Items.Extension – this item template adheres to our custom Extension class standard. This will prefix your file and class name with Ext.
    • EGCS.CRM.Items.Entity – (coming soon) this is our custom standard entity class that all entity classes should follow.
Guidelines

.NET

For classes, properties / fields, and methods, please provide comments for them using the XML comments built into Visual Studio. You can easily insert XML comments by typing triple slash ///. Provide as much inline comments as you think is necessary, and will help other developers read and take over your code.

JavaScript

JavaScript Standards (Documentation). See JavaScript documentation guide

Coding Style

As we do not wish to be too rigid and force developers to adopt a specific coding style, please write your code adhering to Microsoft's guidelines for .NET. There may be some conventions to follow, and more may appear in the future.

Namespace

A single namespace for the entire shared project, which is named after the name of the shared library. For example, EGCS.CRM.Plugins will have a namespace with the same name. There may be special namespaces within each shared project in the event that we want developers to explicitly reference. A common use is to not pollute existing classes with extension methods like System.Linq unless desired.

CRM SDK

When you are referencing CRM SDK's in your project, please use the NuGet packages provided by Microsoft. You can find the CRM SDK NuGet packages under the username crmsdk. This keeps your SDK's up to date and not lost on your local machine.

Documentation

There are two types of documentation: user documentation and API (code) documentation.

User Documentation

This documentation is to help users install, use, and troubleshoot the CRM extensions we develop. An CRM extension can be a plugin, a custom workflow activity, a Web Resource, and many others.

API Documentation

This type of documentation is to ensure that our code is documented properly, so that whoever takes over can easily get on track. It is also to document the shared libraries so that developers know what code is available to them and how it works. We want to eliminate boilerplate code as much as possible.

Code Organization

To ensure that every developer knows exactly where to retrieve, update, and store code, please adhere to the following organization structure of EGCS's code base.

Version Control

Currently at EGCS the version control used is Team Foundation Server (TFS).

Structure

The following is the basic structure for all server side Visual Studio projects. Add custom folders to your project as you see fit, but try to place code in these folders when appropriate.

  • Actions These are early bound classes of CRM custom actions. Name these classes using the custom action's logical name.
  • Base These are base classes that are fundamental to the shared library.
  • Constants They are to be prefixed with Const, as in ConstPluginMessages.
  • Entities These are early bound classes of CRM entities. Name these classes using the entity's logical name.
  • Enumerations They are to be prefixed with Enum, as in EnumPluginStages.
  • Exceptions Custom exceptions are stored here. Please suffix your exception class with the word "Exception".
  • Experimental Any code in these folders are experimental and not yet officially contributed to the shared library. They will be reviewed by all members of the development team. Use at your own risk.
  • Extensions They are to be prefixed with Ext, as in "ExtIOrganizationService. There are two types of extensions: extension method classes and derived classes. Their have their own respective folder. For extension method classes, please declare them as partial classes so that they can be extended by others or organized into smaller files.
  • Helpers These are helper classes that aid another class. Sometimes we may want to extend classes but not directly, so these are intermediate helper classes.
  • Interfaces These are custom interfaces that are prefixed with I, as in IOrganizationService.
  • Library These classes are usually self-contained libraries that performs a particular function, such as the concatenator class.
CRM Extensions

Please refer to the following sections on where to store your code on TFS. Please check in your code under the following location on TFS:

  • Plugins As a standalone Visual Studio Project: EGCS.CRM.Framework\Projects\Plugins
  • Custom Workflow Activities As a standalone Visual Studio Project: EGCS.CRM.Framework\Projects\Workflows
  • Web Resources As web resource files, such as .html, .js, .css, etc. under the appropriate section: EGCS.CRM.Framework\Projects\WebResources
  • Tools As a standalone Visual Studio Project under one of the following locations:
    • For console applications: EGCS.CRM.Framework\Projects\Tools\Console
    • For rich client applications: EGCS.CRM.Framework\Projects\Tools\Client
Shared libraries

To eliminate boilerplate code, promote code reuse, and maintain consistency, we have developed a shared library, the EGCS CRM Framework if you will, that is to be used, maintained, and contributed to by everyone on the development team.

This is also to ensure that the logic behind common functionality is consistent, sound, and easy to troubleshoot. Otherwise, everyone's version of, say, AJAX works differently and some versions may be more or less efficient, more or less error prone, or handle different (edge) cases correctly / incorrectly. This makes it difficult to pinpoint errors in code.

For the shared library itself, we are using Shared Projects introduced in Visual Studio 2015. Shared Projects are different from a normal Visual Studio Project. It is just a container for files (code, assets, etc.) and nothing else; however, a special property of shared project is compiling the referenced shared project directly into the project's assembly / exe file and folder. This eliminates the use of ILMerge on assemblies, which can sometimes cause problems.

EGCS.CRM.NET

This is the general server side shared library. Code that can be used by all server-side extensions should be placed here. The following are the components that make up this shared library. More can be added or split up as we progress.

EGCS.CRM.Plugins

This is the main shared library for plugin development. It contains the plugin base and context as well as some other useful classes for plugin development.

EGCS.CRM.Plugins.Extensions.Registrator

This shared library includes the optional auto plugin step registration feature that can be added to your plugin if desired. Use the EGCS.CRM.Plugins.AutoPlugin template to have it set up for you.

EGCS.CRM.Tools

This shared library includes code that is valuable to both CRM client and console applications.

EGCS.CRM.Tools.Client

This shared library includes code specific to CRM client applications.

EGCS.CRM.Tools.Console

This shared library includes code specific to CRM console applications.

EGCS.CRM.Workflows

This is the main shared library for custom workflow activity development. It contains the workflow base and context as well as some other useful classes for workflow development.
