Financial integration entities

The entities in the table below are used to integrate GCPM data with your organization's financial system.

Entities for financial integration

Entity Name

Description

FI Funding Case

Stores project information. e.g. advances outstanding, hold back available, net disbursed.

FI VendorCustomer
  • The vendor customer entity creates and updates vendor and / or customer information from GCPM and the financial system.
  • Configurators have flexibility to assign this record both a vendor and customer.
  • For organizations that track vendors separately from customers, create a single vendor customer record to hold both the vendor and / or customer information for a given account

FI Commitment

FI Commitment Detail
  • FI Commitment creates, updates and holds commitment information from the financial system. FI commitment has a one to many relationships (1:N) to FI Commitment Details.
  • For example, commitment lines with different attribute data (e.g. fiscal year and coding block) are managed through the FI Commitment Detail entity.
  • FI Commitment acts as a container to commitment lines.
  • When the commitment is sent to the API the FI Commitment and its details are sent together in the payload.

FI Payment

FI Payment Detail
  • The FI Payment entity is designed to create, update and hold financial system payments in GCPM. It has a relationship to FI Payment Detail entity.
  • For payments that have one or more lines, detail records are created.
  • When the payment is sent to the API the FI Payment and its details are sent together in the payload

FI Receivable

FI Receivable Detail
  • The FI Receivable entity creates receivables and their details in the Financial System from GCPM. Receivable details are initially derived from the repayment schedule in GCPM.
  • After initial creation, FI Receivables and their details will only update the financial system with amendments to the repayment schedule.
  • Any other details, such as interest or charges, will not originate from GCPM – instead, the financial system takes precedent and GCPM will update the receivable records periodically from reading the financial system.

FI PAYE

FI PAYE Detail
  • This entity is used to set up a PAYE object in the financial system from GCPM.
  • No components were built beyond the initial submission of the PAYE.
  • Note: GCPM did not ship with the ability to update a PAYE from GCPM to the financial system. This would need to be developed as part of your organization's localization.

  • This entity has a relationship with all parent FI entities (not the detail entities), with the exception of coding block.

Coding Block Configuration

Coding Block 1-10

Coding Block Configuration | PG

Coding Block Configuration | PJ
  • The coding block functionality consists of a Coding Block Configuration and Coding Block 1, 2, 3 … up to 10.
  • The coding block entities contain coding block information of an organization.
  • FI Coding Block Configurations are connected to a program and project via intersect entities.
  • Coding block data is sent with any entity that references it, but no automation has been built to populate these coding blocks, as this is unique to your organization.

FI User

This entity stores financial system-related information about the user triggering a given request.

FI Transaction Log
  • The transaction log entity captures all requests and responses from the financial system API.
  • This entity has a relationship with all FI entities (with the exception of coding block).
  • All FI workflow steps that communicate with the financial system directly write data to this entity (i.e.: submission or retrieval of data).

FI Responsibility Centre
  • This entity stores information about your organization's responsibility centres.
  • You may wish to rename this entity if your organization's name for this differs.
