About amendments

Amendments safeguard the integrity of contribution agreements (CAs) over time. The amendment feature locks / unlocks the GCPM fields that appear in a CA. Both formal and administrative amendments can take place during CA negotiation and project administration stages.

GCPM includes a number of pre-configured amendment types (e.g. administrative amendment, substantive 2-way amendment, default, etc.). Each type of amendment has a set of related transitions, validations and approval paths. When a user chooses a type of amendment, GCPM displays all amendable fields needed on the amendment form.

Amendment categories group appropriate amendment items together. For example, a cost amendment includes tabs with amendable items related to project costs, cash flow, repayment schedule and financing. This list of amendment categories can readily be modified by your organization.

An amendment does not overwrite project data until approval is granted for the changes, ensuring data integrity. All amendable items are shown based ont he project assessment, and display a locked "actual" (or current) version of the data, beside the amendable version. Upon approval, data is automatically copied to the project. No additional steps are required by users.

Amendments are captured with a snapshot and displayed on the project amendment grid.

Your organization can:

  • Choose when amendments can occur, based on the project life cycle for any given program.
  • Choose who can initiate amendments.
  • Modify the approval authority, validations, document generation, etc. associated with amendments, to fit your requirements.

Category Details

Key project data
  • Project-specific data can be amended (e.g. costs, cash flow, dates, etc.).
  • GCPM provides a list of common amendable fields.
  • Configuring additional amendable fields requires using a number of configuration tools (clone, grids, lock, validations, etc.).

Amendment objective
  • A configurable grid is provided for RDAs to capture amendment objectives to facilitate reporting.
  • No reporting tool has been configured.

Amendment decision
  • Decisions related to the configured amendment transitions are readily accessible.
  • Configure decisions to match your organization's processes using the transition tool.
Design considerations:
  • Amendments are managed in a separate entity, and data is copied back and forth between amendment and project entities.
  • There are no saved copies of amendments. Once an amendment is finalized, a snapshot is created.

  • The clone plug-in copies amended data into the appropriate project fields. Every amendable field must be configured with the clone plugin.

Backlog / future design

  • Currently there is no automated mechanism to prevent users from amending data in a tab, then de-selecting the tab so it does not appear on the form.

  • When an amendment is completed, a new CA must be manually generated. It is not auto-generated.

  • Amendments can only modify information in an associated project, not in a client record.
