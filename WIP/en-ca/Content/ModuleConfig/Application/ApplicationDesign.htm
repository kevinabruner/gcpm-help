Application design

The portal is configured to display forms to portal users. The forms can display data that is stored on the portal, or linked to a specific program or project in GCPM. When configuring the application module, consider how the application data submitted by clients will be collected, and how to best leverage automation, report on data, score applications, and determine eligibility.

If you are using SmartGuide to generate portal forms

All application forms are Smartlets (Smartlets are also known as forms), but not all Smartlets are Applications.

SmartGuide uses static and dynamic components, to define how to populate the information the user sees:

  • Static fields are hard coded onto a Smartlet using the SmartGuide designer and exist when the Smartlet is initialized. Drag and drop the field onto the form, use the naming convention <entity>.<field>. The field will exist on the form whenever a user views it, unless business rules (or other program logic) hides or removes the field. Typically static fields collect "Tombstone" data such as name, address and other information that is required from any applicant, for any program.
  • Dynamic components as users enter information about their project and organization, the data presented to the user changes dynamically. Questions can be tailored to client input.Note that responses are captured in GCPM in a grid view, which limits some functionality associated with response data. See Dynamic portal forms. Dynamic components exists in multiple forms. For example: 
    • Application form: When a user selects a program, the Smartlet queries CRM for the appropriate application template that contains application questions. These questions are returned as JSON to the form, and the form uses the metadata in the JSON returned to create a new set of fields (for the questions) on the forms. These fields did not exist on the form when it was initialized. Users see unique versions of the application, depending on the program selected.
    • Advance and Claim form: In these forms, the FC Profile ID is sent to CRM, to retrieve the budget categories associated with the project. These categories dynamically populate the existing budget category dropdown in the Advance / Claim Item grid.
    • Performance Measurement Report The Smartlet queries CRM for the FC Profile Id and creates N rows in the Outcomes grid, where N = the number of outcomes for the project. The JSON returned in this call populates read only labels in each row of the grid.

Design considerations

  • Applications can incorporate generic and dynamic data. The examples in GCPM MVP can inform design decisions for local application intakes.
  • Duplicate and invalid applications can be identified using automated tools.
  • Applications are converted to project records, which can be assessed.
  • Applications submitted by clients:
    • In the User Zone (on the portal) ARE NOT automatically associated with accounts and GC clients. There are many variables to consider with applications submitted by non-validated users. Your organization will need to carefully consider procedures to accommodate changes to account / GC client data, when the information submitted via portal is different from existing records. For example, a work flow can be run on applications on demand (individually or in bulk), to create accounts, GC clients, and project numbers from these applications.
    • In the Client Zone ARE automatically associated with GC clients.
  • Note: changes to certain data fields may impact legal agreements already in place.

Use the following questions to guide designing applications for your organization.

How will you use the data you are collecting?

The current logic behind the application design allows for tombstone data to be collected from generic fields (created on the application entity in GCPM, and on the Alphinat portal form). GCPM functionality can be leveraged to use the data captured in these fields, such as reporting, lookups, etc.

Dynamic data is captured using application questions in the GCPM Application Template and displayed on the Alphinat portal form. The imported information is displayed in the dynamic Questions and Responses section within the Applicant Responses grid in the Application entity.

What information does your organization require for all programs? 

Applications can be designed to collect "tombstone" (or generic) data from static fields. These fields exist on the application entity, and are exposed in an Alphinat portal form. Use GCPM data capture features such as reporting, lookups, etc. Having common generic data collected across programs will simplify your application design requirements. See Generic application questions

  • Static questions are stored on the application entity in CRM.
  • See Generic application questions for more information.

What information does your organization require that is specific to a program?

The program that a client selects can determine which dynamic questions appear on the application form. Portal users can enter data about their project or organization that dynamically determines how information is presented to the user (when using Alphinat portal forms). Consider how to design the information flow to make it easy for the client to provide all information required to assess their application.

The program that the client selects is mapped to a funding opportunity. This determines which program-specific questions appear on the application form.

Do you want to score applications automatically?

You can rank applications to quickly triage high-volume application intake. Rapid triage facilitates pipeline development for timely and informed decision making. Priority scores can be generated automatically for applications submitted on the portal. For example, applications that meet priority sectors can get a higher score. In this case, use of dynamic form is optimal.
See Configure automated priority ranking for applications

Do you want to control which programs clients can select?

The Alphinat portal form can be responsive to information the client enters about their project and organization. Data presented to the user dynamically changes based on these responses. The examples in GCPM demonstrates the dynamic functionality based on the program the client selects can determine which dynamic questions appear on the application form. The static questions appear on all applications, regardless of program selected.
