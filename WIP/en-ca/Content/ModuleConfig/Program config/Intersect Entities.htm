About intersect entities

Intersect entities are used to create an N:N (many-to-many) relationship between two entities. For example, in order to attach a list of Disbursement Questions to a Disbursement Checklist Template, the intersect entity Disbursement Checklist Template Element is used . This allows the same question to be used on more than one template and for a template to have more than one question.

Intersect entities have a 1:N relationship with at least 2 entities, thus connecting them. For example, Disbursement Checklist Template Element, has a lookup to Disbursement Question and a lookup to Disbursement Checklist Template.

Naming intersect entities

Intersect entities, like any entity, must have a name. The intersect entity only exists for the purpose of connecting records, so you should consider using the Calculated field plugin to automatically name these records. Consider using an MBUN (meaningless but unique identifier) which can be generated using the Sequence generator plugin. Using lookup values in the name is not recommended: lookup values are volatile - e.g. they can change without the intersect record's name being changed, potentially getting out of date and causing confusion).

If you must use the lookup value (because this intersect record is being displayed to the user):

  • Ensure the intersect records are updated with the latest values in an automated fashion, or produce an exception report that is monitored regularly.
  • Create a date field on each of the connected entities, then create a workflow that updates the date fields any time the Name field is updated.
  • Create a report that checks whether the intersect fields' modified date is less than the dates on the referenced records.

Dynamics CRM also allows for the creation of N:N relationships natively, but there are pros and cons to each approach. These are outlined in the table below.

Approach

 Pro Con

Custom intersect entity

Allows for complete control including the ability to create additional fields (metadata).

Ability to display both sides of the relationship on one form. For example: on the Account a grid for "Accounts related to me" and a grid for "Accounts I'm related to".

Entity store becomes cluttered with these utility entities.

Requires automated naming that is either meaningless or volatile.

Out of the box intersect entity (also known as an association)

Clean interface, these entities are internal and completely hidden.

No need to figure out automated naming.

No control, cannot add additional metadata. (This is often the deal breaker between Custom and OOB).

Cannot display both sides of the relationship in a grid on a form.

Requires code for any kind of bulk data editing.
