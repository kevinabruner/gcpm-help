Authentication for RDA users

An RDA user is a type of account available for the portal. RDA users are staff members of the department, not members of the general public. When a staff member needs to view the contents of a portal site, they should log into the portal as an RDA user.

An RDA user does not use GCCF (GCKey / SecureKey) to authenticate for the portal. An RDA user is authenticated in one of two ways:

  • Windows authentication: The RDA user provides domain credentials for the Active Directory domain which the Portal belongs to, or a domain which is trusted by the portal’s domain. This login is handled directly by SharePoint and IIS.
  • ADFS server: This is an ADFS server distinct from the ADFS server for GCCF. This ADFS server authenticates against an Active Directory domain chosen by the department.

While it is technically possible for a department to allow both Windows authentication and ADFS authentication for RDA users, in practice it is expected each department will choose one method for staff members to access the portal.

An RDA user is still a SharePoint user. RDA user access to client sites is dependent on SharePoint security roles, similar to any other user. An RDA user cannot access a Client Zone to which he / she does not have permissions.

Similar to GCCF User, RDA users may belong to client sites in one of these roles:

  • Read: The user can read content and secure messages on the site.
  • Edit: Read access, PLUS upload and edit files, and edit forms.
  • Admin: Edit access, PLUS: 
    • Invite new GCCF users to the client using the invitation web part.
    • Use the User Management web part to change the role of GCCF users in the client, or remove users from the client.
    • An RDA user with Admin privilege can add to or remove other RDA users from the client.

RDA users are treated differently than GCCF users in the portal code, and have different privileges and restrictions than GCCF users. These differences are:

  • A GCCF user cannot add or remove an RDA user from a client site. GCCF users cannot manage staff members.
  • An RDA user is allowed to edit forms, but cannot submit forms. Submitting forms is a function reserved for GCCF users (the actual clients of the department).

    Note, it is possible to override this restriction in the portal configuration. However it should only be done in a development or testing environment. Not production.

  • An RDA user cannot send secure messages from the portal. They can only send secure messages from within GCPM.
  • The portal does not automatically copy an RDA user’s profile to SharePoint. If it is desirable to have RDA user names and email addresses available on the Portal, the department should provision SharePoint’s User Profile service.
  • The portal does not present RDA users with a list of clients they belong to. It is expect that staff will use GCPM to find client sites on the portal.
  • RDA users are not invited using the invitation web part. The user management web part has a tab which allows RDA users to be directly added to a site by email address (the GCPM server must be able to resolve that email address into an account name).
  • There is no restriction on using PowerShell or other code to add or remove RDA users from SharePoint sites. The portal does not track RDA users the way it tracks GCCF users.

Technical Details

  • If the portal uses ADFS to authenticate RDA users, a trusted identity token issuer will be provisioned to connect to that ADFS Server. The token issuer can be retrieved via PowerShell (assuming the provider is named “rda-adfs”):

    Get-SPTrustedIdentityTokenIssuer -Identity "rda-adfs"

  • An RDA user will have an identity claim similar to one of these (exact encoding will vary from installation to installation).

    Authenticated via Windows authentication:

    i:0#.w|gscs\whitep

    Authenticated via ADFS:

    i:0ȃ.t|rda-adfs|whitep

  • The portal code base has helper methods to identify RDA users.

    GC.GCCF.SharePoint.Connector.SecurityManager.PortalUserTypeGet() == PortalUserType.RDA

    or

    GC.GCCF.SharePoint.Connector.SecurityManager.PortalUserTypeGet() == PortalUserType.Windows

    Those calls will check if the current user is an ADFS authenticated or Windows authenticated RDA user (suggested to always check both). Internally, the portal code will treat both of those as RDA users.

  • There are other methods in available to check a user’s type and work with RDA users. See the class GC.GCCF.SharePoint.Connector.SecurityManager.
  • When provisioning Client Zones and Project Zones, the portal has the ability to automatically grant access to groups of RDA users. This saves having to individually add RDA Users to sites. These groups of RDA users are identifier by claims.

    For example: On the integration portal, a claim to identify all users who logged in via an RDA ADFS server:

    c:0!.s|trusted%3arda-adfs

    A claim to identify all users who logged in via an RDA ADFS server, who belong to group “gscsportaladministrators”:

    c:0-.t|rda-adfs|gscsportaladministrators

Suggestion

  • Inspect the set of claims for an RDA user. Either use log entries written to GCCF.dbo.GccfLog, or use the debugging web page https://<portal>/_layouts/15/gccf/debug.aspx  
  • Note that debugging needs to be turned on in the gccf.config file. See GC.GCCF configuration.
