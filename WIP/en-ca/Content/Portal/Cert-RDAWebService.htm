RDA web service and portal web service certificates

The RDA web service and the portal web service are certificate-authenticated, WCF web services. Although these web services are accessed over unencrypted HTTP, WCF uses certificates to authenticate the caller and the server, and encrypt all messages.

  • It is not necessary to use commercial certificates for these web services.
  • Certificates can be created using a tool such as Open SSL.
  • Certificate can be issued from a department’s existing Certificate Authority.

In order for the portal to call the RDA web service, or for the RDA to call the portal web service, both sides must be correctly configured.

Server configuration:

  • The CA certificate, without private key, should be in Trusted Root Certification Authorities.
  • The web service should have a valid certificate with a “server” profile.
  • The web service server certificate, with private key, should be in the Local Machine Personal store on the server.
  • The service account for the web service must have “read” access to the server certificate.
  • Web.config must contain the root CA thumbprint.
  • Web.config must list the thumbprints of authorized clients.
  • Web.config must reference the server certificate thumbprint in <serviceCertificate>

Client configuration:

  • The CA certificate, without private key, should be in Trusted Root Certification Authorities.
  • The client should have a valid certificate with a client profile.
  • The client’s certificate, with private key, should be in the Local Machine Personal store.
  • The service account for the client making the web service call much have “read” access to the client certificate private key.
  • The client’s web.config, or egcs.config must contain the root CA thumbprint.
  • The client’s web.config, or egcs.config must contain the client certificate thumbprint to be used when calling the server.
  • The client’s web.config, or egcs.config must contain the server’s certificate thumbprint.

Security

To ensure the security of web service communications, and to protect against unauthorized use of web services, it is essential that the private keys of all certificates are protected. Anyone who has access to these certificates can impersonate calls to the web services, or decrypt web service traffic.

  • Access to certificates must be restricted to trusted personnel.
  • Access to CA files must be restricted to trusted personal, if creating a CA for the portal.
  • Files must be stored in a location which is access-controlled. They should not be kept on a public drive share, for example.

Note that certificates which have been exported without private keys do not need to be treated as sensitive.

Example configuration

See the EGCS.Portal project folder Certificates for an example of using OpenSSL to create a CA and issue certificates. This was the configuration used to generate all certificates for the integration portal environment.

Example commands to load certificates into the Local Machine Personal Store:

$password = Get-Credential -UserName "Enter password for cert" -Message "Enter Password for cert"

Import-PfxCertificate -Exportable -Password $password.Password -CertStoreLocation Cert:\LocalMachine\My -FilePath .\<certificate>.p12

Or

certutil.exe -importpfx .\<certificate>.p12

Example command to grant a service account "read" access to a certificate private key (change the search command on the first line to any preferred comparison to locate the certificate, such as searching by Subject).

$cert = Get-Item cert:\\LocalMachine\My\ | Where-Object { $_.Thumbprint -eq "<thumbprint>" }

$privateKeyContainer = $cert.PrivateKey.CspKeyContainerInfo.UniqueKeyContainerName

$keyPath = "C:\ProgramData\Microsoft\Crypto\RSA\MachineKeys\$privateKeyContainer"

$acl = Get-Acl -Path $keyPath

$permission = "<domain>\<service account>","Read","Allow"

$accessRule = new-object System.Security.AccessControl.FileSystemAccessRule $permission

$acl.AddAccessRule($accessRule)

Set-Acl $keyPath $acl

Example command to load a root CA certificate into Trusted Root Certificate Authorities:

Import-Certificate -CertStoreLocation Cert:\LocalMachine\Root\ -FilePath .\<Root CA>.crt
