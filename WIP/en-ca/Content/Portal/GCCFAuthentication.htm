Authentication for GCCF users

A GCCF User is type of user account available for the portal. GCCF users are internet users who are authenticated to the portal through the GCCF infrastructure. This means they have logged in using GCKey, or Sign-In Partners (SecureKey).

  • It is assumed that GCCF users are clients of the department (members of the general public), and not staff members. Staff members should not use GCCF user accounts.
  • GCCF users automatically have access to any area of the portal which is set to “Anonymous”, or where read access is permitted to the Everyone claim “c:0(.s|true” such as:
    • The root site collection (/)
    • The Welcome Centre (/welcome-bienvenue)

GCCF users may be invited to client sites using the invitation web part. This web part will send an email. Logging in to the portal and accepting the invitation will add a GCCF user to the target client site.

Note that the invitation web part sends an email, but the user accepting the invitation may choose to log in with any GCCF user account they have (or even create a new account when accepting the invitation). You cannot restrict an invitation to a specific GCCF user, it is the person accepting the invitation who chooses the GCCF user account.

GCCF users may be removed from a client site using the User Management web part.

GCCF users belong to a client site in one of three roles:

  • Read: The user can read content and secure messages on the site. They cannot upload or edit files, or forms, or send secure messages.
  • Edit: The user can read content and secure messages on the site. They can upload files and edit files, edit and submit forms, and send secure messages.
  • Admin: The user has all of the Edit privileges. They can also use the Invitation web part to invite new GCCF users to the client. The can use the User Management web part to change the role of other GCCF users in the client, or remove those users.

A record of which GCCF users belong to which client sites is stored in the GCCF database table GCCF.dbo.Invitation. This record is updated by the invitation web part, and the User Management web part.

GCCF users are SharePoint users. Their permissions are controlled by portal code which assigns them to a predefined set of portal security groups.

A profile for each GCCF user is stored in the GCCF database. The GCCF user’s name and email address are automatically synced to each client they belong to.

GCCF users must not be managed using PowerShell or the SharePoint object model. They should only be managed using portal web parts (user management, invitation) or by calling portal web services, or portal user management code.

The portal code ensures that the invitation table stays in sync with the membership of client sites. If the Invitation table is not in sync with actual site memberships, users may be presented with a misleading list of site memberships or secure messages.

To be clear—the Invitation table does not control what sites a GCCF user belongs to. SharePoint permissions control that. Creating a row in Invitation will never grant a user access to a site, nor will removing a row remove a user from a site. However the master page, and various other pieces of the portal consult the invitation table as a cache of which sites a user should belong to, and will present the user site names or secure message counts based on that data.

Technical details

  • A GCCF user is a user whose identity claim is issued by an ADFS server which is joined to the GCCF environment.
  • Internally, SharePoint provisions a Trusted Identity Token Issuer to accept users authenticated by the GCCF ADFS server. The token issuer can be retrieved via PowerShell (assuming the provider is named “gccf-adfs”):

    Get-SPTrustedIdentityTokenIssuer -Identity "gccf-adfs"

  • A record of every GCCF user is stored in the table GCCF.dbo.GccfPai.

    This table records the unique identifier string issued by GCKey / SecureKey and maps it to a GUID. That GUID is known as the “PortalUserId”. The PortalUserId is exclusively throughout the portal code when passing the identity of a GCCF user.

  • When a GCCF user creates a profile, their profile is stored in the table GCCF.dbo. egcs_PortalUser. This table also stores the URL of the user’s personal site, which is the Personal Zone of that user.
  • The user’s personal site records the PortalUserId of the assigned user in a property bag. <SPWeb>.AllProperties["egcs_PortalUserId"] contains a string representation of the PortalUserId.
  • A GCCF user will have an identity claim similar to this (although the exact encoding will vary from installation to installation). The highlighted part of the identity claim is the PortalUserId.

    i:0ǵ.t|gccf-adfs|2969e80c-01bc-4f23-81a3-4ac9c59f1fa7

  • The portal codebase offers a method to determine if the current user is a GCCF user.

    GC.GCCF.SharePoint.Connector.SecurityManager.PortalUserTypeGet() == PortalUserType.GCCF

  • Code to retrieve the PortalUserId of the current user (assuming the current user is a GCCF user):

    using GC.GCCF.SharePoint.Connector;

    ((IClaimsIdentity)HttpContext.Current.User.Identity).ToPortalUserId();

  • Alternately, there’s an extension method on SPUser (works for any user, not just the SPContext current user):

    • SPContext.Current.Web.CurrentUser.ToPortalUserId();

  • There are other methods available to check a user’s type, or obtain their PortalUserId. See the class GC.GCCF.SharePoint.Connector.SecurityManager.
