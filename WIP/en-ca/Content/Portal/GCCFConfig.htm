GC.GCCF configuration

Configuration settings for the GC.GCCF solution are stored as .NET settings in the GCFF Library project. Any environment which loads GC.GCCF.Library.dll must also load configuration settings for the DLL, otherwise the GCCF code will revert to compiled-in defaults, which will almost certainly be wrong.

  • Configuration is typically loaded from web.config, or a .config file for the executable which references the GC.GCCF.Library.dll.
  • In most cases the .config file will contain a reference to GC.GCCF.Properties.Settings:

<configSections>

<sectionGroup name="applicationSettings" type="System.Configuration.ApplicationSettingsGroup, System, Version=4.0.0.0, Culture=neutral, PublicKeyToken=b77a5c561934e089" >

<section name="GC.GCCF.Properties.Settings" type="System.Configuration.ClientSettingsSection, System, Version=4.0.0.0, Culture=neutral, PublicKeyToken=b77a5c561934e089" requirePermission="false" />

</sectionGroup>

</configSections>

An exception to the above is the case of the GCCF Attribute Store, a DLL loaded by ADFS and which loads GC.GCCF.Library.dll. In this case there is no .config file. When GCCF Attribute Store is registered with ADFS, a limited number of settings are also registered with ADFS and used in place of a .config file. These settings relate to the database connection for the GCCF database and logging. The GCCF attribute store uses a carefully-chosen subset of functions in GC.GCCF.Library.dll which does not require other settings.

The settings are stored as XML. They are considered static. They are loaded when a runtime environment using GC.GCCF.Library.dll is initialized, and do not change over the lifetime of execution. The runtime environment must be reloaded to read new settings. This means an IISRESET, application pool reload, or stopping / restarting a service or executable.

Installation

GC.GCCF configuration settings are deployed by a PowerShell script. The script copies the settings to all locations on the current server which require it, and reset / reload any dependant applications.

Installing GC.GCCF configuration requires a build of the GC.GCCF solution, which is assumed to be available in a directory “_Build”.

  1. Install GC.GCCF configuration, and specify the configuration file to be loaded:

    .\_Build\Scripts\Install-GCCF_Config.ps1

  2. The configuration file deploys the following:
  • SharePoint web application for the portal. Modifications are written to web.config for the web application, all zones. The portal web application is recycled in IIS.

    Files:
    C:\inetpub\wwwroot\wss\VirtualDirectories\<folder>\web.config

  • The GCCF Common Domain web application

    Recycles the application pool.

    File: C:\inetpub\wwwroot_gccf_commondomain\cdc

  • The ADFS Extensions web application.

    Recycles the application pool.

    File: C:\inetpub\wwwroot_gccf_adfs\AdfsExtensions

  • The Document Management Windows Service.

    Restarts the service.

    File: C:\Program Files\EGCS\EGCS.Portal.Service.DocumentManagement\gccf.config

  • The Site Creation Windows Service.

    Restarts the service.

    File: C:\Program Files\EGCS\EGCS.Portal.Service.SiteCreation\ gccf.config

  • The Portal web service.

    Recycles the application pool.

    File: C:\inetpub\wwwroot-webservice-http\EGCS.Portal.WebService.Portal\gccf.config

Configuration – Master Configuration File

The master configuration file maps servers to environment names. It allows PowerShell installation scripts to select the correct environment when installing GC.GCCF components. The file is located:

GC.GCF\Configuration EGCS-SESC\Config\config.xml

Add lines to this file to map servers or domains to an environment name. Lines are tested in order, the first matching server name or domain name selects the environment.

E.g.:

<Map Server="PWHITE-SPS4" Environment="phillip" />

<Map Domain="GsCs.local" Environment="integration" />

<Map Domain="ACOA-APECA.GC.CA" Environment="integration" />

A GC.GCCF installer script run on the server “PWHITE-SPS4” will select environment “phillip”.

An installer script run on a server in the domain “GSCS.LOCAL” or “ACOA-APECA.GC.CA” will select environment “integration”.

Configuration – Environment Configuration File

The environment configuration file is loaded when a PowerShell installer reads config.xml and selects an environment. The Environment Configuration File defines most of the settings needed to install GC.GCCF components for a specific environment.

The Environment Configuration File is located:

GC.GCF\Configuration EGCS-SESC\Config\<environment>.config.xml

The “Token” section defines most of the configuration values which need to be changed for a Portal. The Token section contains name / value pairs which are substituted into the rest of the file, and into other configuration files. The substitution pattern is %Token%, e.g.: %PortalName%.

New tokens can be added to the token section as desired.

Configuration – gccf.config Application Settings

Application settings are the .NET settings for the GC.GCCF.Library application. Effectively this is the contents of app.config. This configuration data is copied into SharePoint’s web.config, or becomes the gccf.config file for other executables or web applications.

These settings define the runtime values used by GC.GCCF code.

The gccf.config file is located here: GC.GCF\Configuration EGCS-SESC\Config\GccfLibrary\<environment>.gccf.config
