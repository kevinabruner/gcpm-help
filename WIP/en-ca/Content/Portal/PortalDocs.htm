Documents on the portal

On the portal, documents are stored in SharePoint document libraries. See Portal document integration with GCPM  for a graphic of the architecture.

Note:

  • All document libraries have versioning enabled. Uploading a new copy of a document, or editing a document will produce a new version.
  • Document libraries are protected by SharePoint site permissions. Generally speaking, users will have the same access to a document library that they have to the zone. If a user has been invited with Read access to a Client, they will have Read access to all document libraries under that client or any project of the Client. If a user does not have access to a client, they will not have access to any document library under the client, or any project of the client.
  • Documents may be uploaded by the SharePoint web interface, or by dragging and dropping files onto the browser window while visiting shared documents.
  • Document libraries support a folder structure.
  • Documents deleted from a document library go to the SharePoint recycle bin. Portal users may browse and recover documents from the recycle bin in the case of accidental deletion.
  • SharePoint supports integration with Microsoft Office Online Server (OOS). If SharePoint is linked to an OOS server, users will be able to edit Office documents (e.g. Word, Excel) directly in their web browser. This is an optional feature. If OOS is not present, users will have to download, edit, and re-upload documents to make changes.
  • Editing documents using OOS creates new versions. Re-uploading a document will also create a new version, so long as the file name has not been changed.

  • URLs for the various document libraries are configured in EGCS.Portal configuration.

    • See EGCS.Portal configuration. Note that changing URLs in the configuration only affects new sites—existing sites will not be affected. However, the portal code will attempt to use the new URL names to locate document libraries which will result in broken links for existing sites.

    • URL names should be decided before any Client, Project, or Personal zones are created on the Portal, and not changed afterwards.

Document libraries

Document libraries are pre-created by the Windows Service – Site Creation See Windows service - site creation

Each document library has a specific purpose:

  • Shared Documents: Arbitrary documents uploaded by portal users to share with the RDA, or other portal users of the same client or project. Word documents, spreadsheets, pictures, or any other type of document.

    This document library is visible on the portal master page. The user always has a link to for shared documents which is contextual for the current zone (Client, Project, or Personal).

    EGCS.Portal configuration setting - DocumentLibraryPortalDocument

    Default URL - https://<portal>/<site>/Document

  • Submissions: Alphinat SmartGuide forms, and files “attached” to those form submissions. A folder is automatically created in this library for each form created by portal users. The XML file for the SmartGuide form, and all attached files for that form are stored in the form’s folder.

    Once a form has been submitted, security permissions on the form’s folder are changed to make all contents read-only for all users. This prevents any further changes to a form or file attachments.

    There is no link on the portal’s master page to this document library. It is visible when editing SmartGuide forms, and using the file attachment function. The document library is not hidden, there is simply no convenient link to navigate to it. If a user is aware of the document library URL, they can choose to navigate to that library.

    EGCS.Portal configuration setting - DocumentLibraryPortalSubmission

    Default URL - https://<portal>/<site>/SubmissionSoumission

  • Records: Snapshots, letters, or other formal records available for display to the portal. These documents originate at the RDA and are copied to the Portal. This document library is read-only for all users of the Portal.

    There is no link on the portal master page to this document library. Various web parts display links to snapshots and other records as needed. The document library is not hidden, there is simply no convenient link to navigate to it. If a user is aware of the document library URL, they can choose to navigate to that library.

    EGCS.Portal configuration setting - DocumentLibraryRdaRecord

    Default URL: https://<portal>/<site>/Record

Document synchronization with GCPM

The Windows service – document management (see Windows service - document management) periodically scans both the portal and GCPM for new or updated documents. These documents are copied from the portal to GCPM, or from GCPM to the portal, as required.

When interfacing with GCPM the portal always and only talks to the RDA web service (see RDA web service ). It uses methods in the DocumentManager.svc service to copy documents back and forth. The RDA web service is then responsible for copying documents into GCPM.

The DocumentManager.svc web service talks to GCPM via a set of public API methods when synchronizing documents.

Portal submissions are handled by the FormManager.svc web service, which uses a separate set of public API methods to submit the data and attachments to GCPM.

See GCPM portal components for information on the public API methods.

Documents in GCPM

In GCPM, documents are stored in the internal document repository, which is most likely either GCDocs or SharePoint.

  • Documents in GCDocs can move around freely and may not be in its original uploaded location. SharePoint does not have this ability without breaking the link to the document.
  • Document versioning is handled by the internal document repository.

  • Any document from the portal has two copies:

    1. The official copy stored in GCPM’s internal document repository, and
    2. The convenience copy stored in SharePoint on the portal.

  • Documents in GCPM that originated from the portal can only be modified on the portal and not in GCPM. Any modifications made directly to the portal documents in GCPM will not be reflected on the portal, i.e. the copies in GCPM are read-only.

  • GCPM uses the custom-built Document Manager plugin for all interactions between GCPM and the internal document repository.

  • The following document location configurations must be configured for the entities: account, portal user, and project. The name of the folder is not important.

    • Portal Documents: this folder stores any documents uploaded to the portal by the client.

    • Portal Submissions: this folder stores any documents related to a form submission in the portal, e.g. submission data, submission snapshot, submission attachments, etc.

  • RDA Record documents are synced to the portal whenever a document in GCPM is flagged as a portal document AND is associated to either an account, portal user, or project.
  • Before a client or project portal site is created, officers in GCPM can pre-flag documents as portal documents which will sync to the portal when the portal site is created.
  • When a document from the portal is synced over, GCPM will replicate the folder structure of that document. This prevents documents with the same name from overwriting each other.
