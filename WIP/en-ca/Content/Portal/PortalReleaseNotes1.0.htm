Portal release notes 1.0

The GCPM portal is more than a SharePoint farm. It is composed of multiple components that work together, from ADFS, CRM, to Alphinat SmartGuide, to custom web applications. It is important that each of these components be understood and planned for before installation begins. An installation of the portal is not complete unless all components are accounted for.

Planning is essential to successfully deploy a portal. Attempting to jump into the installation phase without understanding the components, where they are installed, and how they communicate, will be an exercise in frustration.

Requirements / prerequisites

  • Microsoft SharePoint 2016 on-premises. The number of servers and topography of the SharePoint farm is up to the department.
    See Sharepoint installation for more detailed discussion.
  • Microsoft SQL Server supported by SharePoint 2016
  • Microsoft ADFS 2016 (requires Windows Server 2016)
  • Optional: Microsoft Office Online Server, if the portal is to support in-browser editing of documents.
  • Alphinat SmartGuide, current version. The GCPM portal team has worked with Alphinat to integrate fixes and improvements into SmartGuide to support forms as they are used on the portal.
  • A reverse proxy server:
    • If the SharePoint farm has multiple Web Front Ends (WFEs), the reverse proxy must support “sticky sessions” or “session affinity”. The reverse proxy must be able to route all web request from a client to the same SharePoint WFE.
      If the SharePoint farm has only a single WFE, this requirement is waived.
  • The reverse proxy must be able to direct traffic based on HOST header, as well as URL.
    E.g.: Requests for https://<portal>/abc can be routed to a different back-end server than https://<portal>/123

    Any reverse proxy implementation which meets those restrictions is acceptable. The reverse proxy function could be implemented by a hardware device, a firewall, or a Windows or Linux server. The GCPM portal team used Microsoft’s free Application Request Routing package during the development of the GCPM portal.

  • The department must be registered to use the GCCF infrastructure (GCKey and SecureKey). The department must have ICM certificates for their portal, and have provided a valid Portal metadata package to GCCF. This is arranged through SSC.
  • DNS names must be selected and registered for the Portal, ADFS, and the GCCF Common Domain.
  • Firewall rules must be configured to pass inbound traffic to the reverse proxy server.
  • SSL certificate(s) for all public Portal DNS names (Portal, ADFS, GCCF Common Domain).
    See document “SSL Certificate”.
  • Certificates to authenticate the RDA web service and portal web service. These can be generated using OpenSSL, issued by a department’s CA, or purchased as commercial certificates.
  • A topography plan, that identifies the servers that will be used for the portal and where they will be located. Communication channels between the servers must be understood and accounted for.

Topography

Topography considers how the components of the portal are deployed onto servers. The traffic routing rules required on the reverse proxy server will be defined by these choices.

The reference implementation is the GCPM integration portal used during development. This implementation used separate VMs for most portal components.

Notes:

  • It is recommended to keep the reverse proxy as a separate server. This adds an extra layer of isolation between portal servers and the Internet. The reverse proxy terminates and routes all incoming web traffic.
  • The GCCF ADFS server can likely be combined with the application server to host Alphinat SmartGuide. ADFS is only accessed momentarily as users are logging on, or logging off. The overall load on ADFS is extremely light.
  • The GCCF ADFS server and GCPM ADFS Server must be separate from each other. It is not possible to combine them.
  • The GCPM application server is not necessarily hosted in the same data centre as other portal components. It should be hosted logically “close” to GCPM CRM to minimize latency, as those web services have to interact closely with CRM.
  • A production SharePoint deployment should use a separate SQL Server. Do not install SQL Server onto the same server as SharePoint.
  • An extreme deployment could see all portal components piled onto one server. This configuration has not been tested, and is not recommended.
