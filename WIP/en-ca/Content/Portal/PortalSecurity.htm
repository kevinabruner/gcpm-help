Portal security

Security is an important topic for the portal. Various components of the portal have different security requirements. Care must be taken with each component to minimize the risk of intrusion, or accidental data disclosure.

Specific components of the portal with specific security considerations:

Reverse Proxy

Security considerations

  • The reverse proxy server is directly connected to the internet.

  • The reverse proxy should be the only server which is directly available from the internet. All other servers should be isolated from the internet.

  • Standard anti-virus and intrusion detection software, as per department policies, should be installed.

  • The reverse proxy should block attempts from internet sources to perform Windows authentication.

Authentication

  • None. All web requests from the internet are unauthenticated (anonymous). Even a GCCF-authenticated connections to the portal will appear to be anonymous, so far as the reverse proxy is concerned.

GCCF Database

Security considerations

  • As a best practice for SQL Server, “SQL server accounts” (username and password) should not be enabled. Only domain authentication should be supported.
  • As a best practice for SQL Server, the well-known “sa” account should not be enabled.

Authentication

  • Portal components authenticate via their Windows service accounts.
  • Read and write access to table in the GCCF database is only supported via stored procedures.
  • All service accounts accessing the GCCF database should belong to a role, such as GCCFAdfs or GCCFUser.
SharePoint

Security considerations

  • GCCF Users will be authenticated by an ADFS server dedicated to GCCF.
  • The department needs to choose how RDA users will be authenticated:
    • By Windows authentication, using domain credentials.
    • By an ADFS server for the RDA (not the same ADFS server used for GCCF).
  • The department needs to decide if GCCF Users and RDA Users will access the portal using the same URL. Will everyone use https://portal.gc.ca, or do RDA users use https://rdaportal.gc.ca instead?
  • If all users access via the same url, that SharePoint authentication zone must support both GCCF and RDA authentication providers.
  • It is strongly recommended that Windows authentication not be permitted from the internet. This exposes all accounts in the RDA’s active directory domain to potential remote access. Even if https://portal.gc.ca permits Windows authentication, the reverse proxy or firewall should be configured to block such authentication from the internet.
  • Access to zones is granted via SharePoint security groups and roles. A set of SharePoint security groups on each site determines the access level of users.
  • Every Client zone and Personal zone is implemented as a separate SharePoint site collection. Only users who require access to a client should be added to the Client Zone site collection. Only the single user who is assigned to a Personal Zone should be granted access to the site collection.
  • Project zones are SharePoint sites, and inherit their security from Client Zones.
  • Private consideration: assume that users who belong to a site collection can see the names and email addresses of other users who belong to the same site collection. E.g.: all users within a Client Zone can see other users of that Client Zone.
  • Users should never been added directly to the root site, or the Welcome Centre site collections.
  • The Portal Invitation process manages the roles of users in Client sites, and synchronizes with information in the GCCF database. Don’t manage GCCF Users with the SharePoint UI, or PowerShell. Only use portal web parts and web services.
  • It is the department’s responsibility to ensure that permissions are configured correctly for Client Zones and Personal Zones. Misconfiguration can lead to a security breach or data exposure. The portal ’s code cannot protect from misconfiguration of permissions.
    • Anonymous access must never be granted to Client Zones or Personal Zones.
    • The “Everyone” claim c:0(.s|true must never be granted access to Client Zones or Personal Zones.

Authentication

  • Anonymous for the root site of the portal.
  • By ADFS for GCCF Users for Client and Personal Zones, and the Welcome Centre.
  • By Windows authentication or ADFS for RDA users.
  • In all cases, Windows authentication should be blocked from the internet. SharePoint cannot restrict this, it must be configured in the reverse proxy or firewall.
SharePoint web services

Security considerations

  • SharePoint web services are hosted by, and authenticated by SharePoint.

  • A SharePoint web service runs with the identity of the caller.

  • A SharePoint web service is accessed from a relative URL in a SharePoint site.

    /<site>/_layouts/15/EgcsPortal/<web service page>

  • SharePoint web services are the primary interface for portal users to trigger behaviours in the portal, this includes:

    • Sending and accepting invitations

    • Managing users roles and site membership

    • Editing and submitting forms

  • For the most part, SharePoint security is used. Web services perform SharePoint operations using the context of the caller. If the caller does not have access to read or write a file, neither will the web service.

  • Execution is elevated to the SharePoint service account for operations that must be performed using that account: calling the RDA web service, or accessing the GCCF database.

  • Care must be taken to ensure that elevation is only performed if the web service caller is authorized to perform a certain operation. The SharePoint web service uses custom security code to validate user’s permissions for tasks such as issuing site invitations or managing user roles in sites.

  • A custom class PortalWebServiceMessageInspector validates every call to SharePoint web services. It ensures Referrer and Origin headers are acceptable for the portal. In most cases it blocks attempts by anonymous users (unauthenticated users) to call SharePoint web services.

  • The EGCS.Portal setting WebServiceAnonymousMethods lists a set of web services and methods which are permitted to be called anonymously. Only methods which are absolutely required for anonymous users should be added to this list.

Authentication

  • SharePoint web services share authenticated by SharePoint. Web services are called with the identity of the current SharePoint user.
The Portal web service

Security considerations

  • Should run under the SharePoint service account

  • Should be run with an application pool service account which has access to the GCCF database (automatically true for the SharePoint service account).

  • Must not be exposed to the internet. For internal use of portal components only.

  • Communicates over HTTP, but all messages are encrypted and signed by WCF using certificates.

Authentication

  • Via service certificates only.

  • The portal web service should have a unique server certificate.

  • Any RDA components which need to call the portal web service (such as the WebAPI web service) should have a unique client certificate.

RDA web service

Security considerations

  • Must run under a service account which can make connections to GCPM CRM which are authorized to access all entities needed to support the portal.
  • Alternately, must have credentials suitable use with Microsoft’s CRM SDK to make connections to GCPM CRM which are authorized to access all entities needed to support the portal.
  • May require credentials to access legacy data sources at the RDA.
  • Communicates over HTTP, but all messages are encrypted and signed by WCF using certificates.

Authentication

  • Via service certificates only.
  • The RDA web service should have a unique server certificate.
  • The portal should have a unique client certificate.

The WebAPI web service

Security considerations:

  • Currently no authentication is required.
  • May wish to implement OAuth2 authentication to secure the endpoint.

Authentication

  • No authentication required.
  • If OAuth2 is used, portal plugin in GCPM requires the client id and client secret key.
  • Portal web service calls are authenticated via service certificates.
  • The Web API web service should have a unique server certificate.
  • The portal should have a unique client certificate.
GCCF Common Domain web application

Security considerations

  • Should be run with an application pool service account which has access to the GCCF database.

Authentication

  • None. All access to this web application is anonymous.

ADFS Extensions web application

Security considerations

  • Should be run with an application pool service account which has access to the GCCF database.
  • Must be able to connect to the memcached service on all SharePoint web front-ends.

Authentication

  • None. All access to this web application is anonymous.
Memcached

Security considerations

  • Must not be exposed to the internet.
  • Only needs to accept connections from the ADFS Extensions web service.

Authentication

  • None
