About the decision framework

GCPM Decisions is a flexible CRM feature that allows your organization full configuration and control on how to record decisions in the system. The Decision Framework is an extension of the About the transition framework, used to automate steps taken when a decision is reached in a transition. To implement the Decision Framework on an entity, there must be a separate entity to save the decisions. The naming convention for the separate entity is [ENTITY] Decision.

When a Decision Type is configured for a transition, the transition executes the decision and creates a decision record in the [ENTITY] Decision entity. One or more documents will be generated according to the Decision Type settings.

Every action an employee makes as a function of his / her job can be considered a decision - so it is up to your organization to determine what should be captured as:

  • Operational:  administrative or routine decisions, which do not require a record of each person who has recommended / approved a decision
    Examples: assessment finalized, claim review completed)
  • Statutory or formal: decisions related to delegated authority that require a record indicating who was involved in the decision and the action of each (e.g. approve / deny).
    Examples: project approval indicates who recommended the project for approval, and the choice of the final authority.

Decision documents

Decision framework has the functionality to provide links to documents (in a Sharepoint or GCDocs environment) that have been linked to the decision. These documents may be system-generated snapshots of relevant information available to the decision-maker, or documents uploaded by a person involved in the decision. The information related to these documents is saved in the Document entity.

Document records are attached to the decision, not to the entity: e.g., attached to PAYE Decision, not PAYE. All the documents linked to a decision are listed in the Related Documents grid in an [Entity] Decision form.

System generated documents are configured in the related Decision Type record.

  • For more information on how these documents are generated, see About the document generator.
  • For information on how to predefine the location to upload these documents, see About the document manager solution.

A link to manually added documents in the Related Documents grid is created based on the document's location.

How to generate decisions

To implement the decision framework, a Decision Type must be set in a transition path record. If an entity follows that path in a business process flow, a decision will be documented. The last person in a recommendation generates a decision.

  • Decisions that do not require approval:
    • These can be generated without any links to the transition framework. An example of this is the Application Decision. These decisions are automatically generated on the user's behalf via workflow and have no linkage to the transition framework.
    • If you wish to have a decision entered by the user as part of a transition, you can set the Decision Type field in the Transition Path
  • Decisions that require approval: 
    • To take advantage of the approval functions you will need to use a transition. A Decision Type and Approval type must be set in the applicable transition path record. If an entity follows that path in a business process flow, a decision will be documented. The last person in a recommendation generates a decision.

Decisions that trigger actions

The decision framework can interact with the action framework to automate actions for certain decisions. For example, a decision can be automated to change the status of a workflow, send an email, or create a task.

A Transition Path record creates the reference between a Decision Type and the action sets to be used. For information on actions sets triggered on decisions, see About the transition framework.

For information on how to configure action sets, see About the action framework.
