Document manager connectors

Connectors are responsible for authenticating the user against and making calls to an external Document Management System (DMS).

The document manager plugin determines which DMS connector to use solely on the "Document Repository" option set found in the CFG Document Entity entity, using the option value to dynamically call the correct DMS connector.

Out of the box, the options are mapped as follows:

For example, selecting the SharePoint option will make the plugin call the custom action egcs_dm_Execute1 and use the egcs_Username1 field, which are mapped to and picked up by the SharePoint connector plugin.

Notes:

  • If you change the mappings, ensure the corresponding custom action is associated to the correct connector plugin.
  • If you extend the plugin to support other external DMS, add a new option to the Document Repository option set and create the corresponding custom action and user name field (in the systemuser entity) with the same naming convention and parameters.
SharePoint

SharePoint’s REST API is used to make calls to SharePoint. The calls are authenticated by sending the user name and password of the service account to SharePoint to retrieve the SAML token, FedAuth cookie, and Form Digest, which are not cached.

Because the document manager plugin is an external application to SharePoint that uses its client-side API, it cannot authenticate or impersonate as the calling user without the user’s user name and password, so the use of an admin service account is required. CreatedBy and ModifiedBy fields will consequently be tagged with the service account.

Developers can override the plugin’s SharePoint authentication mechanism, to locally implement caching, user impersonation or direct user authentication. The SharePoint connector can also be overridden if developers choose to use SharePoint’s alternate APIs, such as SharePoint’s Client Side Model Object (CSOM).

See Document manager plugin API for more information.

GCDocs

GCDocs uses OpenText for its document management system, which provides a SOAP-based web service endpoint for external applications to use. The plugin authenticates to GCDocs using the provided user name and password of an admin service account and proceeds to impersonate as the calling user for all subsequent calls to GCDocs.

Certain calls are made under the service account to retrieve data that the user may not have access to; however, the data retrieved from these calls are never returned to the user and any calls that modify GCDocs are made under the calling user. This way, user permissions are honoured and users are prevented from performing actions that the calling user would not normally be able to perform in GCDocs.

There is only one way to authenticate and connect to GCDocs programmatically, so no overrides are provided.
