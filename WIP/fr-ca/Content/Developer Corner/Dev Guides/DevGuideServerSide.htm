Guide de développement du côté serveur

Veuillez noter que les éléments suivants sont sujets à des modifications, idéalement avec un avis adéquat. Si vous n'êtes pas sûr d'une des lignes directrices ci-dessous, veuillez demander à votre chef d'équipe de plus amples renseignements.

Comment commencer

Veuillez télécharger et utiliser nos modèles EGCS CRM comme point de départ pour tous vos travaux de développement. Cela assurera que le projet respecte les lignes directrices établies ci-dessous.

Développement de EGCS

Veuillez télécharger les modèles EGCS CRM sur les serveurs du WD et commencer votre développement à partir des modèles installés.

Modèles actuellement disponibles :
  • Modèles de solution EGCS.CRM.Framework Téléchargez tout d'abord cette solution si vous n'avez pas la bibliothèque partagée sur votre appareil.
  • Modèles de projet
    • EGCS.CRM.Plugins.SimplePlugin – this sets up a simple plugin project based on our custom plugin base, plugin context, and logging capabilities.
    • EGCS.CRM.Plugins.AutoPlugin– this sets up the same project as the simple plugin but includes the auto plugin step registration feature installed. You will have to configure the auto plugin step registration for it to know how to create your plugin steps.
    • EGCS.CRM.Workflows.SimpleWorkflow - cela établit un projet d'activité de flux personnalisé simple basé sur notre base de flux personnalisé et sur notre contexte de flux.
  • Modèles d'élément
    • EGCS.CRM.Items.Constant – this item template adheres to our custom Constant class standard. This will prefix your file and class name with Const.
    • EGCS.CRM.Items.Enumeration – this item template adheres to our custom Enumeration class standard. This will prefix your file and class name with Enum.
    • EGCS.CRM.Items.Extension - ce modèle d'élément respecte notre classe standard Extension personnalisée. Cela ajoutera un préfixe à votre ficher et au nom de classe qui est Ext.
    • EGCS.CRM.Items.Entity – (à venir) c'est notre classe standard d'entité personnalisé que toutes les classes d'entité doivent suivre.
Lignes directrices

.NET

Pour les classes, les propriétés, les champs et les méthodes, veuillez offrir des commentaires en utilisant les commentaires XML intégrés dans Visual Studio. Vous pouvez aisément insérer des commentaires en entrant trois barres obliques ///. Donnez autant de commentaires entre les lignes que vous pensez qu'il est nécessaire et cela aidera les autres développeurs à lire et à prendre en charge votre code.

Style de codage

Puisque nous ne souhaitons pas être trop rigide et ne nous voulons pas forcer les développeurs à adopter un style de codage particulier, veuillez écrire votre code en respectant les lignes directrices pour .NET. Il peut y avoir des conventions à respecter et d'autres peuvent se rajouter à l'avenir.

Espace de nommage

Un espace de nommage unique pour un projet partagé entier, lequel est nommé selon le nom de la bibliothèque partagée. Par exemple, EGCS.CRM.Plugins will have a namespace with the same name. There may be special namespaces within each shared project in the event that we want developers to explicitly reference. A common use is to not pollute existing classes with extension methods like System.Linq à moins que souhaité.

CRM SDK

Lorsque vous faites référence à un CRM SDK dans votre projet, veuillez utiliser les ensembles NuGet offerts par Microsoft. Vous pouvez trouver les ensembles NuGet de CRM SDK sous le nom d'utilisateur crmsdk. Cela maintient votre SDK à jour et qu'il n'est pas perdu sur votre appareil local.

Documentation

Il y a deux types de documentation: la documentation de l'utilisateur et la documentation de l'API (code).

Documentation de l'utilisateur

Cette documentation sert à aider les utilisateurs à installer, utiliser et dépanner les extensions de CRM que nous développons. Une extension  CRM peut être un plugiciel, une activité de flux personnalisée, une ressource Web et bien d'autres.

Documentation de l'API

Ce type de documentation s'assure que notre code est documenté de manière appropriée, afin que la personne qui prend la relève peut facilement continuer le travail. Elle sert aussi a documenter les bibliothèques partagées afin que les développeurs puissent savoir quel code leur est disponible et de quel façon il fonctionne. Nous voulons éliminer le code passe-partout le plus possible.

Codage de l'organisation

Afin de s'assurer que chaque développeur sait exactement où récupérer, mettre à jour et stocker le code, veuillez respecter la structure d'organisation suivantes du code de base du EGCS.

Contrôle des versions

Actuellement au EGCS, le contrôle des versions utilisé est le Team Foundation Server (TFS)

Structure

Ce qui suit est la structure de base pour tous les projets côtés serveurs de Visual Studio. Ajoutez des dossiers personnalisés à votre projet quand bon vous semble, mais essayez de placer des codes dans ces dossiers lorsque c'est approprié.

  • Actions -  Ce sont les classes liés préalablement des actions personnalisés de CRM. Nommer ces classes en utilisant le nom logique de l'action personnalisé.
  • Base - Ce sont les classes de base qui sont fondamentales dans la bibliothèques partagée.
  • Constantes - Elles doivent avoir le préfixes Const, comme dans ConstPluginMessages.
  • Entités -  Ce sont les classes liés préalablement des entités de CRM. Nommer ces classes en utilisant le nom logique de l'entité
  • Énumérations - Elles doivent avoir le préfixes Enum, comme dans EnumPluginStages
  • Exceptions - Les exceptions personnalisées sont stockées ici. Veuillez placer un suffixe à voter classe d'exception avec le mot « Exception ».
  • Expérimental - Tous les codes dans ces dossiers sont expérimentaux et ne font pas partis officiellement de la bibliothèque partagée. Ils seront examinés par tous les membres de l'équipe de développement. Utilisez à vos propres risques.
  • Extensions - Elles doivent avoir le préfixes Ext, comme dans « ExtIOrganizationService ». Il y a deux types d'extension: les classes de méthode d'extension et les classes dérivées. Ils ont leur dossier respectif. Pour les classes de méthode d'extension, veuillez les déclarer comme classes partielles afin qu'elles soient étendues aux autres ou organisées dans de plus petits fichiers.
  • Aidantes - Ce sont des classes aidantes qui appuient une autre classe. Parfois, nous voudront peut-être étendre des classes, mais pas de manière directe, ce sont donc des classes aidantes intermédiaires.
  • Interfaces - Ces sont des interfaces personnalisées qui possèdent le préfixe I, comme dans IOrganizationService.
  • Bibliothèque - Ces classes sont généralement des bibliothèques autonomes qui effectuent une fonction particulière, comme la classe concaténatrice
Extensions de CRM

Veuillez vos référer aux sections suivantes sur l'endroit où stocker votre code sur le TFS. Veuillez vérifier votre code sous l'emplacement suivant sur le TFS :

  • Plugiciels - Comme un projet indépendant de Visual Studio :EGCS.CRM.Framework\Projects\Plugins
  • Activités de flux personnalisées - Comme un projet indépendant de Visual Studio : EGCS.CRM.Framework\Projects\Workflows
  • Ressources Web - Comme fichiers de ressources Web, comme .html, .js, .css, etc. sous la section appropriée :EGCS.CRM.Framework\Projects\WebResources
  • Outils - Comme un projet indépendant de Visual Studio sous l'un des emplacements suivants :
    • Pour les applications de console :EGCS.CRM.Framework\Projects\Tools\Console
    • For rich client applications: EGCS.CRM.Framework\Projects\Tools\Client
Bibliothèques partagées

Afin d'éliminer le code passe-partout, promouvoir la réutilisation de code et maintenir la consistance, nous avons élaborer une bibliothèque partagée, le EGCS CRM Framework (CADRE EGCS CRM) si vous la voulez, qui sera utilisée, entretenue et contribué à tous à les membres de l'équipe de développement.

C'est aussi pour s'assurer que la logique derrière la fonction commune est consistante, logique et facile à dépanner. Autrement, la version de tous de, par exemple, AJAX fonctionne différemment et certaines versions peuvent être plus ou moins efficace, moins ou plus sujette à des erreurs ou gère différents cas de manière correcte ou incorrecte. Cela rend difficile de déterminer les erreurs dans le code.

Pour la bibliothèque partagée, nous utilisons des projets partagés introduits dans Visual Studio 2015. Les projets partagés sont différents d'un projet Visual Studio normal. C'est seulement un contenir pour les fichiers (code, biens, etc.) et rien d'autre; par contre, une propriété spéciale d'un projet partagé est la compilation de projet partagé référencé directement dans l'assemblage du projet et de son fichier et dossier exe. Cela élimine l'utilisation de ILMerge sur les assemblages, lesquels peuvent parfois causer des problèmes.

EGCS.CRM.NET

C'est la bibliothèque partagée générale du côté serveur. Le code qui peut être utilisé par toutes les extensions du côté serveur devraient se situer ici. Les éléments suivants sont les composantes qui composent cette bibliothèque partagée. Plus d'éléments peuvent être ajoutés ou séparés alors que nous progressons.

EGCS.CRM.Plugins

C'est la principale bibliothèque partagée pour le développement de plugiciel. Elle contient la base et le contexte de plugiciel ainsi que d'autres classes utiles pour le développement de plugiciel.

EGCS.CRM.Plugins.Extensions.Registrator

Cette bibliothèque partagée comprend la fonction facultative d'enregistrement automatique des étapes de plugiciel qui peuvent être ajoutées à votre plugiciel si désiré. Utilisez le modèle EGCS.CRM.Plugins.AutoPlugin pour qu'il soit installer pour vous.

EGCS.CRM.Tools

Cette bibliothèque partagée comprend des codes qui sont utiles pour les applications des client CRM et des consoles.

EGCS.CRM.Tools.Client

Cette bibliothèque partagée comprend des codes particuliers aux applications des client CRM.

EGCS.CRM.Tools.Console

Cette bibliothèque partagée comprend des codes particuliers aux applications des consoles.

EGCS.CRM.Workflows

C'est la principale bibliothèque partagée pour le développement d'activité de flux personnalisée. Elle contient la base et le contexte de flux ainsi que quelques autres classes utiles pour le développement de flux.
