Conception de la demande

Le portail est configuré pour afficher les formulaires aux utilisateurs de portail. Les formulaires peuvent afficher les données qui sont conservées dans le portail ou liées à un programme ou un projet particulier dans GPSC. Lors de la configuration du module de demande, considérez la façon dont les données de demande présentées par les clients seront recueilles, et de quelle façon on peut le mieux tirer parti de l'automatisation, de la déclaration de données, des demandes de cotes et pur déterminer l'admissibilité.

Si vous utilisez SmartGuide pour générer des formulaires de portail.

Tous les formulaires de demande sont des Smartlets (les Smartlets sont aussi connu sous le nom de formulaires), mais tous les Smartlets ne sont pas des Demandes.

SmartGuide utilise des composantes statiques et dynamiques pour définir la façon pour remplier les renseignements que l'utilisateur voit :

  • Les Champs statiques sont figé dans le code par un Smartlet en utilisant le concepteur SmartGuide et ils existent lorsque le Smartlet est amorcé. Glissez-déposez le champ dans le formulaire et utilisez la convention d'appellation <entity>.<field>. Le champ existera sur le formulaire des qu'un utilisateur le consulte, à moins que les règles opérationnelles (ou une autre logique de programme) le cache ou enlève le champ. Généralement les champs statiques recueillent des données « de base » comme le nom, l'adresse et d'autres renseignements qui est requis de tout demandeur pour tous les programmes.
  • Les Composantes dynamiques font référence au renseignements entrés par les utilisateurs à propos de leur projet et de l'organisation, les données présentées aux utilisateurs changent de manière dynamique. Les questions peuvent être conçues selon les contributions du client. Noter que les réponses sont saisies dans le GPSC dans un tableau, lequel limite quelques fonctions associées avec les données de réponse. Consulter Les formulaires dynamiques du portail. Les composantes dynamiques existent sous de multiples formes. Par exemple :
    • Formulaire de demande : lorsqu'un utilisateur sélectionne un programme, les requêtes Smartlet de CRM pour le modèle de demande approprié qui contient les questions de demande. Ces questions sont retournées comme JSON au formulaire et le formulaire utiliser les métadonnées dans le JSON retourné afin de créer un nouvel ensemble de champs (pour les questions) sur les formulaires. Ces champs n'existent pas sur le formulaire lorsqu'il a été amorcé. Les utilisateurs voient des versions uniques de la demande selon le programme sélectionné.
    • Advance and Claim form: In these forms, the FC Profile ID is sent to CRM, to retrieve the budget categories associated with the project. These categories dynamically populate the existing budget category dropdown in the Advance / Claim Item grid.
    • Performance Measurement Report The Smartlet queries CRM for the FC Profile Id and creates N rows in the Outcomes grid, where N = the number of outcomes for the project. Le JSON retourne dans cet appel pour remplir les étiquettes en lecture seule dans chaque rangée de la grille.

Points à considérer pour la conception

  • Les demandes peuvent intégrer des données générales et dynamiques. Les exemples dans le MVP de la GPSC peuvent orienter les décisions liées à la conception pour les admissions locales de demandes.
  • Les demandes en double et invalides peuvent être identifiées au moyen d'outils automatisés.
  • Les demandes sont converties en enregistrements de projet, qui peuvent être évalués.
  • Les demandes présentées par les clients :
    • Dans la Zone de l'utilisateur (sur le portail) elles NE SONT PAS automatiquement associées aux comptes et aux client SC. Il y a de nombreuses variables à considérer avec les demandes présentées par des utilisateurs non validés. Votre organisation devra étudier attentivement les procédures afin de tenir compte des changements au compte ou aux données de client de SC, quant les renseignements soumis au moyen du portail diffèrent des enregistrements existants. Par exemple, un flux peut s'exécuté sur des demandes au besoin (individuellement ou en gros) afin de créer des comptes, des clients SC et des numéros de projet à partir de ces demandes.
    • Dans la Zone de l'utilisateur elles SONT automatiquement associées aux client SC.
  • Il faut noter que les changements à certains champs de données pourraient avoir une incidence sur le accords juridiques déjà en place.

En utilisant les questions suivantes pour guider les demandes de conception pour votre organisation.

De quelle utiliserez-vous les données que vous recueillez?

La logique actuelle derrière la conception de demande permet aux données de base d'être recueillies à partir des champs généraux (créé sur l'entité de demande dans GPSC et sur le formulaire du portail Aphinat). La fonction du GPSC peut être mise a profit et afin d'utiliser les données saisies dans ces champs, comme les déclarations, les recherches, etc.

Les données dynamiques sont saisies en utilisant les questions de demande dans le Modèle de demande de GPSC et affichée sur le formulaire du portail Alphinat. Les renseignements importés sont affichés dans la section dynamique des Questions et réponses à l'intérieur de la grille de réponses du demandeur dans l'entité de demande.

De quels renseignements votre organisation a-t-elle besoin pour tous les programmes?

Les demandes peuvent être conçues pour recueillir des données « de base » (ou générales) des champs statiques. Ces champs existent sur l'entité de demande et sont présentés dans un formulaire du portail Alphinat. Utiliser les fonctions de saisie de données du GPSC comme les déclarations, les recherches, etc. Avoir des données générales communes recueillies dans l'ensemble des programmes simplifiera les exigences de la conception de votre demande. Voir Questions pour les demandes génériques

  • Les questions statiques sont stockées dans l'entité de demande dans CRM.
  • Pour obtenir de plus amples renseignements, consultez Questions pour les demandes génériques.

De quels renseignements votre organisation a-t-elle besoin pour un programme particulier?

Le programme qu'un client sélectionne peut déterminer quelles questions dynamiques s'affichent sur le formulaire de demande. Les utilisateurs du portail peuvent entrer leurs données à propos de leur projet et de leur organisation qui détermine de manière dynamique la façon dont les renseignements sont présentés à l'utilisateur (lors de l'utilisation de formulaires du portail Alphinat). Considérer la façon dont concevoir le flux de renseignements pour permettre au client de fournir les renseignements requis plus facilement afin d'évaluer leur demande.

Le programme que le client sélectionne est mis en correspondance à une possibilité de financement. Cela détermine quelles questions particulières à un programme s'affichent sur le formulaire de demande.

Voulez-vous noter les demander automatiquement?

Vous pouvez classer les demandes pour trier rapidement le grand volume de demandes admises. Le triage rapide facilite le développement de pipeline pour la prise de décision opportune et éclairée. Le pointage prioritaire peut être générer automatiquement pour les demandes présentés sur le portail. Par exemple, les demandes qui respectent les secteurs de priorité peuvent obtenir un pointage plus élevé. Dans ce cas, l'utilisation de formulaire dynamique est optimale.
Voir Configurer le classement prioritaire automatisé pour les demandes

Voulez-vous contrôler quels programmes les clients peuvent sélectionner?

Le formulaire du portail Alphinat peut répondre aux renseignements que le client entre à propos de leur projet et de leur organisation. Les données présentées à l'utilisateur de manière dynamique change selon ces réponses. Les exemples dans le GPSC démontrent que la fonction dynamique basée sur le programme que le client sélectionne peut déterminer quelles questions dynamiques apparaissent sur le formulaire de demande. Les questions statiques apparaissent sur toutes les demandes, sans égard au programme sélectionné.
