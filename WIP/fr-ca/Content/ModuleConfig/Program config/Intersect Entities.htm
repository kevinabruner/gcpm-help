À propos des entités intersectionnelles

Les entités intersectionnelles sont utilisées afin de créer une relation N:N (de nombreuses à de nombreuses) entre les deux entités. Par exemple, afin de joindre une liste de questions des débours à un modèle de la liste de contrôle des débours, l'entité intersectionnelle Élément du modèle de liste de contrôle de déboursement est utilisée. Cela permet d'utiliser la même question dans plus d'un modèle et pour qu'un modèle ait plus d'une question.

Les entités intersectionnelles ont une relation 1:N avec au moins 2 entités, ce faisant, les reliant. Par exemple, Élément du modèle de liste de contrôle de déboursement a une fonction de recherche pour la questions des débours et une recherche pour le modèle de liste de contrôle de déboursement.

Nommer les entités intersectionnelles

Les entités intersectionnelles, comme toutes entités, doivent avoir un nom. L'entité intersectionnelle peut seulement exister pour relier des registres, vous devez donc envisager d'utiliser le Champ calculé plugiciel afin de nommer automatiquement ces enregistrements. Envisagez d'utiliser un MBUN (numéro unique non significatif) qui peut être produit au moyen du plugiciel de générateur de séquence. Il n'est pas recommandé d'utiliser les valeurs de recherche dans le nom : les valeurs de recherche sont volatiles (p. ex., elles changent sans que le nom de l'enregistrement intersectionnel ne soit changé, devenant possiblement daté et engendrant une confusion).

Si vous devez utiliser la valeur de recherche (parce que cet enregistrement intersectionnel est affiché à l'utilisateur) :

  • Assurez-vous que l'enregistrement intersectionnel est mis à jour de façon automatisée avec les valeurs les plus récentes, ou produisez un rapport d'exception qui est surveillé régulièrement.
  • Créez un champ de date dans chacune des entités reliées, puis créez un flux de travail qui met à jour les champs de date chaque fois que le champ Nom est mis à jour.
  • Créez un rapport qui vérifie si la date modifiée des champs intersectionnels est inférieure aux dates dans les enregistrements de référence.

Dynamics CRM prévoit également la création de relations N:N natives, mais chaque approche comporte des avantages et des inconvénients. Ils sont indiqués dans le tableau ci-dessous.

Approche

 Avantage Inconvénient

Entité intersectionnelle personnalisée

Permet le contrôle complet, y compris la capacité de créer des champs supplémentaires  (métadonnées).

Capacité d'afficher les deux côtés de la relation dans un formulaire. Par exemple, dans le Compte, une grille de comptes qui me sont liés et une grille de comptes auxquels je suis lié.

Le stockage d'entités devient encombrés par ces entités d'utilitaires.

Nécessite un nom automatisé qui est soit dénudé de sens ou volatile.

Entité intersectionnelle automatique (aussi appelée association)

Interface claire, ces entités sont internes et complètement cachées.

Nul besoin de déterminer un nom automatisé.

Aucun contrôle, ne peut pas ajouter de métadonnées supplémentaires. (C'est souvent l'achoppement entre personnalisé et prête à l'emploi).

Ne peut pas afficher les deux côtés de la relation dans une grille dans un formulaire.

Nécessite un code pour tout type de modification de données en lot.
