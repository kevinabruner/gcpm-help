Authentification pour les utilisateurs des agences de développement régional (ADR)

Un utilisateur d'ADR est un type de compte disponible dans le portail. Les utilisateurs d'ADR sont des membres du personnel du ministère, et non des membres du grand public. Lorsqu'un membre du personnel doit visionner le contenu d'un site de portail, il devrait ouvrir une session dans le portail à titre d'utilisateur d'ADR.

Un utilisateur d'ADR n'utilise pas la fédération des justificatifs du gouvernement du Canada (FJGC) (CléGC ou SecureKey) pour s'authentifier dans le portail. Un utilisateur d'ADR est authentifié d'une de deux façons :

  • Authentification Windows : L'utilisateur d'ADR fournit les justificatifs de domaine pour le domaine Active Directory auquel le portail appartient, ou encore un domaine fiable du portail. Cette ouverture de session est prise directement en charge par SharePoint et IIS.
  • Serveur des services ADFS (Active Directory Federation Services) : Il s'agit d'un serveur ADFS qui est distinct du serveur ADFS de la FJGC. Ce serveur ADFS assure une authentification dans le domaine Active Directory choisi par le ministère.

Même s'il est techniquement possible, pour un ministère, de permettre à la fois l'authentification Windows et l'authentification ADFS pour les utilisateurs d'ADR, en pratique, on s'attend à ce que chaque ministère choisisse une seule méthode pour permettre aux membres du personnel d'accéder au portail.

Un utilisateur d'ADR continue d'être un utilisateur de SharePoint. L'accès des utilisateurs d'ADR aux sites clients dépend des rôles de sécurité de SharePoint, comme pour tout autre utilisateur. Un utilisateur d'ADR ne peut accéder à une Zone Client pour laquelle il n'a pas de permissions.

De façon semblable à un utilisateur de la FJGC, les utilisateurs d'ADR peuvent appartenir à des sites clients dans l'un des rôles suivants :

  • Lecture : L'utilisateur peut lire le contenu et les messages sécurisés sur le site.
  • Modification : Accès de lecture, EN PLUS de téléchargement et de modification de fichiers ainsi que de modification de formulaires.
  • Administration : Accès de modification, EN PLUS de ce qui suit :
    • Inviter de nouveaux utilisateurs de la FJGC dans le client à l'aide de la partie Web de l'invitation.
    • Utiliser la partie Web de la gestion des utilisateurs pour changer le rôles des utilisateurs de la FJGC dans le client, ou pour retirer des utilisateurs du client.
    • Un utilisateur d'ADR possédant un privilège d'administrateur peut ajouter d'autres utilisateurs d'ADR dans le client ou en retirer.

Les utilisateurs d'ADR sont traités différemment des utilisateurs de la FJGC dans le code de portail, et ils ont d'autres privilèges et restrictions que les utilisateurs de la FJGC. Ces différences sont :

  • Un utilisateur de la FJGC ne peut ajouter un utilisateur d'ADR à un site client ou l'en retirer. Les utilisateurs de la FJGC ne peuvent gérer les membres du personnel.
  • Il est permis à un utilisateur d'ADR de modifier des formulaires; or cet utilisateur ne peut présenter de formulaires. La présentation de formulaires est une fonction réservée aux utilisateurs de la FJGC (les clients réels du ministères).

    Remarque : Il est possible d'écraser cette restriction dans la configuration du portail. Cela ne devrait toutefois être fait que dans un environnement de développement ou d'essai. Et non de production.

  • Un utilisateur d'ADR ne peut envoyer de messages sécurisés à partir du portail. Il ne peut le faire qu'à partir du GPSC.
  • Le portail ne transpose pas automatiquement dans SharePoint le profil des utilisateurs d'ADR. S'il est souhaitable que le nom d'utilisateur et l'adresse de courriel d'un utilisateur d'ADR soient disponibles dans le portail, le ministère devrait fournir le service de profil d'utilisateur de SharePoint.
  • Le portail ne présente pas aux utilisateurs d'ADR une liste des clients auxquels ils appartiennent. On s'attend à ce que le personnel utilise le GPSC pour trouver les sites clients dans le portail.
  • Les utilisateurs d'ADR ne sont pas invités à l'aide de la partie Web de l'invitation. La partie Web de la gestion des utilisateurs comporte un onglet qui permet d'ajouter directement les utilisateurs d'ADR à un site par leur adresse de courriel (le serveur du GPSC doit pouvoir résoudre cette adresse de courriel en un nom de compte).
  • Aucune restriction ne s'applique à l'utilisateur de PowerShell ou d'autres codes pour ajouter des utilisateurs d'ADR à des sites SharePoint ou les en retirer. Le portail ne suit pas les utilisateurs d'ADR de la même façon qu'il suit les utilisateurs de la FJGC.

Détails techniques

  • Si le portail utilise ADFS pour authentifier les utilisateurs d'ADR, un émetteur de jeton d'identité fiable sera fourni pour effectuer la connexion avec ce serveur ADFS. L'émetteur de jeton peut être récupéré au moyen de PowerShell (en supposant que le fournisseur s'appelle « rda-adfs »).

    Get-SPTrustedIdentityTokenIssuer -Identity "rda-adfs"

  • Un utilisateur d'ADR aura une demande d'identité semblable (le codage exact variera d'une installation a l'autre).

    Authentifié via l'authentification Windows:

    i:0#.w|gscs\whitep

    Authenticated via ADFS:

    i:0ȃ.t|rda-adfs|whitep

  • La base de code du portail dispose de méthodes d'assistance pour identifier les utilisateurs ADR.

    GC.GCCF.SharePoint.Connector.SecurityManager.PortalUserTypeGet() == PortalUserType.RDA

    or

    GC.GCCF.SharePoint.Connector.SecurityManager.PortalUserTypeGet() == PortalUserType.Windows

    Those calls will check if the current user is an ADFS authenticated or Windows authenticated RDA user (suggested to always check both). À l'interne, le code de portail traitera les deux utilisateurs comme des utilisateurs d'ADR.

  • D'autres méthodes sont offertes pour vérifier le type d'utilisateur et le travail d'un utilisateur avec les utilisateurs d'ADR. Voir la classe GC.GCCF.SharePoint.Connector.SecurityManager.
  • Au moment de fournir des zones client et des zones de projet, le portail est en mesure d'accorder automatiquement l'accès a des groupes d'utilisateurs d'ADR. On n'a ainsi pas à ajouter individuellement les utilisateurs d'ADR aux sites. Ces groupes d'utilisateurs d'ADR sont identifiés selon les demandes.

    For example: On the integration portal, a claim to identify all users who logged in via an RDA ADFS server:

    c:0!.s|trusted%3arda-adfs

    A claim to identify all users who logged in via an RDA ADFS server, who belong to group “gscsportaladministrators”:

    c:0-.t|rda-adfs|gscsportaladministrators

Suggestion

  • Inspecter l'ensemble de demandes pour un utilisateur d'ADR. Either use log entries written to GCCF.dbo.GccfLog, or use the debugging web page https://<portal>/_layouts/15/gccf/debug.aspx  
  • Il est à noter que le débogage soit être activé dans le fichier gccf.config. Consulter Configuration de GC.GCCF.
