Certificats de service Web d'agences régionales de développement (ADR) et de service Web de portail

Le service Web d'ADR et le service Web de portail sont authentifiés par certificat, utilisant les services Web de la Windows Communication Foundation (WCF). Même si l'on accès à ces services  Web à l'aide du protocole HTTP sans chiffrement, la WCF utilise des certificats pour authentifier l'appelant et le serveur, et pour chiffrer tous les messages.

  • Il n'est pas nécessaire d'utiliser des certificats commerciaux pour ces services  Web.
  • Les certificats peuvent être créés au moyen d'un outil comme Open  SSL.
  • Un certificat peut être délivré à partir de l'autorité de certification (AC) existante d'un ministère.

Pour que le portail appelle le service Web d'ADR, ou que l'ADR appelle le service  Web de portail, les deux cotés doivent être configurés correctement.

Configuration du serveur :

  • Le certificat AC, sans clé privée, devrait se trouver dans les autorités de certification racines de confiance.
  • Le service Web devrait avoir un certificat valide avec un profil « serveur ».
  • Le certificat de serveur pour service Web, avec une clé privée, devrait se trouver dans le magasin personnel de l'ordinateur local, sur le serveur.
  • Le compte de service pour le service Web doit avoir un accès de « lecture » au certificat de serveur.
  • doit contenir l'empreinte de l'AC racine.Web.config
  • Web.config must list the thumbprints of authorized clients.
  • Web.config must reference the server certificate thumbprint in <serviceCertificate>

Configuration du client :

  • Le certificat AC, sans clé privée, devrait se trouver dans les autorités de certification racines de confiance.
  • Le client devrait avoir un certificat valide avec le profil client.
  • Le certificat du client, avec une clé privée, devrait se trouver dans le magasin personnel de l'ordinateur local.
  • Le compte de service pour le client effectuant l'appel de service doit avoir un accès de « lecture » à la clé privée du certificat du client.
  • Le web.config, or egcs.config , doit renfermer l'empreinte de l'AC racine.
  • Le web.config, or egcs.config , doit renfermer l'empreinte de certificat utilisée au moment d'appeler le serveur.
  • Le web.config, or egcs.config , doit renfermer l'empreinte de certificat du serveur.

Sécurité

Pour veiller à la sécurité des communications du service  Web, et pour se protéger contre l'utilisation non autorisée des services  Web, il est essentiel que les clés privées de tous les certificats soient protégés. Quiconque a accès à ces certificats peut imiter des appels aux services  Web ou déchiffrer le trafic de services Web.

  • L'accès aux certificats doit être limité au personnel de confiance.
  • L'accès aux fichiers d'AC doit être limité au personnel de confiance, si l'on crée une AC pour le portail.
  • Les fichiers doivent être stockés dans un emplacement dont l'accès est contrôlé. Ils ne doivent pas être gardés sur un lecteur public partagé, par exemple.

Veuillez noter que les certificats qui ont été exportés sans clé privée n'ont pas à être traités comme étant sensibles.

Exemple de configuration

Voir le dossier du projet EGCS.Portal Certificats pour un exemple d'utilisation d'OpenSSL pour la création d'une AC et la délivrance de certificats. Il s'agissait de la configuration servant à générer tous les certificats pour l'environnement de portail d'intégration.

Exemples de commandes servant à charger des certificats dans le magasin personnel de l'ordinateur local

$password = Get-Credential -UserName "Enter password for cert" -Message "Enter Password for cert"

Import-PfxCertificate -Exportable -Password $password.Password -CertStoreLocation Cert:\LocalMachine\My -FilePath .\<certificate>.p12

Ou

certutil.exe -importpfx .\<certificate>.p12

Exemple de commande servant à accorder un accès de « lecture » à une clé privée de certificat dans un compte de service (changez la commande de recherche à la première ligne pour toute comparaison préférée destinée à trouver le certificat, comme une recherche par sujet).

$cert = Get-Item cert:\\LocalMachine\My\ | Where-Object { $_.Thumbprint -eq "<thumbprint>" }

$privateKeyContainer = $cert.PrivateKey.CspKeyContainerInfo.UniqueKeyContainerName

$keyPath = "C:\ProgramData\Microsoft\Crypto\RSA\MachineKeys\$privateKeyContainer"

$acl = Get-Acl -Path $keyPath

$permission = "<domain>\<service account>","Read","Allow"

$accessRule = new-object System.Security.AccessControl.FileSystemAccessRule $permission

$acl.AddAccessRule($accessRule)

Set-Acl $keyPath $acl

Exemple de commande servant à charger un certificat d'AC dans des autorités de certification racine de confiance :

Import-Certificate -CertStoreLocation Cert:\LocalMachine\Root\ -FilePath .\<Root CA>.crt
