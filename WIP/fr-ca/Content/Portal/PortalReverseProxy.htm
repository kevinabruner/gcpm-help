Serveur proxy inverse pou le portail

Le portail du GPSC requiert l'utilisateur d'un serveur proxy inverse. Tout serveur de proxy inverse peut être utilisé, matériel ou logiciel, aussi longtemps qu'il respectent deux exigences :

  1. Le serveur doit être en mesure de diriger la circulation basée sur les URL ainsi que sur l'entête de l'HÔTE.

    Par exemple,

    https://portal.dept.gc.ca/url123 can be routed differently than

    https://portal.dept.gc.ca/url456

  2. Si la ferme de SharePoint cible a plus d'une application frontale sur le Web (WFE), le proxy inverse devra supporter les sessions « sticky ». Avec les sessions « sticky », un utilisateur est attribué à une WFE pour la durée de leur session et toute la circulation pour l'utilisateur est envoyée à cette WFE. La circulation de l'utilisateur ne devra pas équilibrer sa charge sur de multiples WFE.

    La mise à l'essai interne a déterminé que SharePoint ne synchronisait pas les renseignements de la session sur de multiples WFE dans certaines circonstances. La meilleure option pour un comportement fiable de la session était de limiter les utilisateurs à une seule WFE.

    Cette exigence peut être ignorer si la ferme SharePoint n'a qu'une seule WFE.

L'ensemble ’ Application Request Routing gratuit de Microsoft est mis à l'essai comme serveur proxy inverse fiable pour ce portail. Par contre, tout proxy inverse qui respectent les deux conditions ci-dessus devraient fonctionner.

Acheminement basé sur les URL

Le portail requiert un acheminement basé sur les URL afin que le SmartGuide d'Alphinat puisse être présenté sur Internet comme s'il faisait parti de SharePoint. Cela permet à SmartGuide de partager le cookie d'identité de l’utilisateur pour l'authentification sur SharePoint.

  • https://portal.dept.gc.ca/smartlets/*- ce trafic est envoyé à SmartGuide
  • https://portal.dept.gc.ca/sgd/*- ce trafic est envoyé à SmartGuide (pourrait ne pas être désirable pour la configuration de la production)
  • https://portal.dept.gc.ca/*- tout autre trafic est envoyé à SharePoint

Un acheminement similaire est utilisé pour les ADFS. Cela permet à l'application Web des ADFS personnalisés d'accéder et d'enlever les cookies définis par les ADFS.

  • https://adfs.dept.gc.ca/adfs/ls/gccf/* - ce trafic est envoyé à l’application Web des extensions Web des ADFS
  • https://adfs.dept.gc.ca/adfs/ls/*- tout autre trafic est envoyé aux ADFS

Remarques et ensemble des règles d’échantillonnage

  • Lors de la mise en cache du trafic pour les ADFS, notez que les ADFS requièrent que le trafic soit chiffré SSL. Il n'est pas possible d'effectuer un déchargement de SS (utiliser le proxy inverse pour mettre fin à la connexion SSL et envoyer un HTTP non-chiffré).
  • D'autres composantes du portail devraient appuyer le déchargement de SSL, au besoin. Par contre, votre organisation aura besoin de mettre à l'essai de manière rigoureuse cette configuration. SharePoint aura besoin de la fonction de Alternate Access Mapping (Mappage des accès de substitution adéquate) pour la configuration du déchargement de SSL.
  • A ruleset for Microsoft ARR is included in the GC.GCCF project. It was extracted from the Integration Portal:

    GC.GCCF\Configuration EGCS-SESC\Config\ARR Configuration\applicationHost.dev.xml

    This is an extract of relevant rules from:

    C:\Windows\System32\inetsrv\config\applicationHost.config

Remarques à propos de l'Environnement d'intégration

  • Le serveur ARR pour l'Intégration a aussi eu l'entrée HÔTE suivante et ces noms d'hôte ont été utilisés dans certaines règles :
    192.168.255.208 egcs-dev.acoa-apeca.gc.ca
    192.168.255.208 egcs-dev-acoa-apeca.fjgc-gccf.gc.ca
  • Pour l'Environnement d'intégration, le serveur des  ADFS hébergeait :
    • ADFS
    • Application Web des extensions des ADFS
    • Application Web du  Domaine commun de la FJGC
  • L'utilisation de fichiers HÔTE et de ces nom d'hôte dans les règles ARR était une solution de rechange pour que ARR négocie les connexions SNI au serveur des ADFS. Ce n'est probablement pas nécessaire pour les autres ministères.

    Dans cette configuration, il y a quatre serveurs :

    • Proxy inverse (exécuter Microsoft ARR)
    • Serveur d'application héberge le SmartGuide d'Alphinat
    • Serveur SharePoint
    • Le serveur des ADFS qui héberge les ADFS, le service Web des extensions des ADFS, le service Web du Domaine commun de a FJGC

    D'autres configurations qui utilisent différents services sur différents serveurs sont possibles.
