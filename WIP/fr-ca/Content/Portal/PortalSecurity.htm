Sécurité du portail

La sécurité est un sujet important pour le portail. Diverses composantes du portail ont différentes exigences en matière de sécurité. Il faut être prudent avec chaque composante afin de minimiser le risque d'intrusion ou de divulgation accidentelle de données.

Les composantes particulière du potai avec des considérations particulière en matière de sécurité :

Proxy inverse

Considérations en matière de sécurité

  • Le serveur proxy inverse est directement connecté à Internet.

  • Le proxy inverse devrait être le seul serveur qui est directement accessible à partir d'Internet. Tous les autres serveurs devraient être isolés d'Internet.

  • Un logiciel standard de détection des virus et des intrusions, conformément aux politiques du ministère, devrait être installé.

  • Le proxy inverse devrait bloquer les tentatives provenant d'Internet d'effectuer une authentification de Windows.

Authentification

  • Aucune. Toutes les requêtes Web d'Internet ne sont pas authentifiées (anonymes). Même une connexion authentifié de la FJGC au portail apparaîtra anonyme, tant qu'il est question du proxy inverse.

Base de données de la FJGC

Considérations en matière de sécurité

  • Une pratique exemplaire pour le serveur SQL les « SQL server accounts » (comptes du serveur SQL) ne devraient pas être activés. Seulement l'authentification par le domaine devrait être supportée.
  • En tant que pratique exemplaire pour le serveur SQL, le compte « sa » bien connu ne devrait pas être activé.

Authentification

  • Les composantes du portail s'authentifient par leurs comptes de service Windows.
  • L'accès lecture-écriture au tableau dans la base de données de la FJGC est seulement appuyé par les procédures conservées.
  • Tous les comptes de service qui accèdent à la base de données e la FJGC devraient appartenir à un rôle, comme GCCFAdfs ou GCCFUser.
SharePoint

Considérations en matière de sécurité

  • Les utilisateur de la FJGC seront authentifiées par un serveur des ADFS consacré à la FJGC.
  • Le ministère a besoin de choisir la façon dont les utilisateurs d'ADR seront authentifiés.
    • Par l'authentification Windows, en utilisant les justificatifs du domaine.
    • Par un serveur des  ADFS pour l'ADR (ce n'est pas le même serveur des ADFS utilisé pour la FJGC).
  • Le ministère a besoin de décider si les utilisateurs de la FJGC et les utilisateur d'ADR auront accès au portail en utilisant le même URL. Est-ce que tout le monde utilisera https://portal.gc.ca, ou est-ce que les utilisateurs d'ADR utiliseront https://rdaportal.gc.ca au lieu?
  • Si tous les utilisateurs ont accès à partir de la même URL, cette zone d'authentification SharePoint doit appuyer les fournisseurs d'authentification de la FJGC et de l'ADR.
  • Il est fortement recommandé que l'authentification de Windows ne soit pas permise à partir d'Internet. Cela expose tous les comptes dans le domaine d'annuaire actif de l'ADR pour l'accès à distance possible. Même si https://portal.gc.ca permet l'authentification Windows, le proxy inverse ou le pare-feu devrait être configuré pour bloquer une telle authentification à partir d'Internet.
  • L'accès aux zones est autorisé par les groupes de sécurité et les rôles SharePoint. Un ensemble de groupes de sécurité SharePoint sur chaque site détermine le niveau d'accès.
  • Chaque Zone client et Zone personnelle st mise en œuvre comme collection de sites SharePoint séparée. Seulement les utilisateurs qui requiert l'accès à un client devrait être ajouté à la collection de sites de la Zone client. Seulement l'utilisateur unique qui est attribué à une Zone personnelle devrait avoir accès à la collection de sites.
  • Les zones de projets sont des sites SharePoint et héritent leur sécurité des Zones client.
  • Facteur privé : il suppose que les utilisateurs qui appartiennent à une collection de sites peuvent voir les noms et les adresses courriel des autres utilisateurs qui appartiennent à la même collection de sites. P. ex. tous les utilisateurs avec une Zone client peuvent voir les autres utilisateurs de cette Zone client.
  • Les utilisateurs ne devraient jamais être ajoutée direction au site racine ou aux collections de sites de Centre d'accueil.
  • Le processus d'Invitation au portail gère les rôles des utilisateurs dans les sites Client et se synchronisent avec les renseignements dans la base de données de la FJGC. Ne gérez pas les utilisateurs de la FJGC avec le SharePoint UI ou PowerShell. N'utilisez seulement que les parties Web et les services Web.
  • Il est de la responsabilité du ministère de s'assurer que les permissions sont configurées correctement pour les Zones client et les Zones personnelles. La mauvaise configuration peut mener à des brèches de sécurité ou à l'exposition de données. Le code du portail ne peut pas protéger la mauvaise configuration des permissions.
    • L'accès anonyme ne doit jamais être accordé aux Zones client ou aux Zones personnelles.
    • La “Everyone” claim c:0(.s|true ne doit jamais être autorisée à avoir accès aux Zones client ou aux Zones personnelles.

Authentification

  • Anonyme pour le site racine du portail.
  • Par les ADFS pour les utilisateurs de la FJGC pour les Zones client et personnelles et le Centre d'accueil.
  • Par l'authentification Windows ou des ADFS pour les utilisateurs d'ADR.
  • Dans tous les cas, l'authentification Windows devrait être bloquer à paprtir d'Internet. SharePoint ne peut pas l'empêcher, il doit être configuré dans le proxy inverse ou le pare-feu.
Services Web de SharePoint

Considérations en matière de sécurité

  • Les services Web de SharePoint sont hébergé et l'authentifié par SharePoint.

  • Un service Web de SharePoint s'exécute avec l'identité de l'appelant.

  • L'accès aux services Web de SharePoint se fait à partir d'un URL relatif au site SharePoint.

    /<site>/_layouts/15/EgcsPortal/<web service page>

  • Les services Web de SharePoint sont l'interface primaire pour les utilisateurs du portail afin de déclencher des comportements dans le portail, cela comprend :

    • Envoyer et accepter des invitations

    • Gérer les rôles des utilisateurs et les membres

    • Modifier et soumettre des formulaires

  • En grande partie, la sécurité de SharePoint est utilisée. Les services Web effectuent les opérations de SharePoint en utilisant le contexte de l'appelant. Si l'appelant n'a pas accès à la lecture ou à l'écriture d'un fichier, le service Web n'aura pas accès lui aussi.

  • L'exécution est élevée au compte de service de SharePoint pour les opérations qui doivent être réalisées en utilisant ce compte : appeler le service Web de l'ADR ou accéder à la base de données de la FJGC.

  • Il faut être prudent afin de s'assurer que l'élévation est seulement effectuée si l'appelant du service Web est autorisé à effectuer une certaine opération. Le service Web de SharePoint utilise un code de sécurité personnalisé afin de valider les permissions de l'utilisateur pour les tâches comme l'émission d'invitation au site ou la gestion des rôles d'utilisateur dans les sites.

  • Une classes personnaliséePortalWebServiceMessageInspector valide tous les appels aux services Web de SharePoint. il s'assure que les entêtes Référant et Origine sont acceptables pour le portail. Dans la plupart des cas, il bloque les tentatives par les utilisateurs anonymes (utilisateurs non authentifiés) pour faire appel aux services Web de SharePoint.

  • Le EGCS.Portal setting WebServiceAnonymousMethods dresse la liste des services Web et des méthodes qui autorisent l'appel anonyme. Seulement les méthodes qui sont absolument requises pour les utilisateurs anonymes devraient être ajoutées à la liste.

Authentification

  • Les services Web de SharePoint partage l'authentification par SharePoint. Les services Web sont appelés avec l'identité de l'utilisateur de SharePoint actuel.
Le service Web du Portail

Considérations en matière de sécurité

  • Il devrait fonctionner sous le compte de service SharePoint

  • Il devrait fonctionner avec un compte de service pour le bassin d'application lequel a accès à la base de données de la FJGC (automatiquement vraie pour le compte de service SharePoint).

  • Il ne doit as être exposé à Internet. Pour l'utilisation interne des composantes du portail seulement.

  • Il communique par HTTP, mais tous les messages ont chiffrés et signés par les WCF en utilisant des certificats.

Authentification

  • Par l'entremise de certificat de service seulement.

  • Le service du portail Web devrait avoir un certificat de service unique.

  • Toutes les composantes de l'ADR qui auraient besoin d'appeler le service Web du portail (comme le service Web de WebAPI) devrait avoir un certificat client unique.

Service Web de l'ADR

Considérations en matière de sécurité

  • Il doit s'exécuter sous un compte de service lequel peut faire des connexions au CRM du GPSC lesquelles ont accès à toutes entités nécessaires pour appuyer le portail.
  • Par ailleurs, Il doit avoir des justificatifs approprié utilisables avec le CRM SDK de Microsoft afin de faire des connexions au CRM du GPSC lesquelles ont accès à toutes entités nécessaires pour appuyer le portail.
  • Il peut avoir besoin de justificatifs pour accéder aux anciennes sources de données à l'ADR.
  • Il communique par HTTP, mais tous les messages ont chiffrés et signés par les WCF en utilisant des certificats.

Authentification

  • Par l'entremise de certificat de service seulement.
  • Le service Web de l'ADR devrait avoir un certificat de service unique.
  • Le portail devrait avoir un certificat de client unique.

Le service Web de WebAPI

Considérations en matière de sécurité :

  • Actuellement aucune considération n'est requise.
  • On peut souhaiter mettre en œuvre une authentification OAuth2 afin de sécuriser le pont de terminaison.

Authentification

  • Aucune authentification n'est requise.
  • Si OAuth2 est utilisé, le plugiciel du portail dans GPSC requiert l'ID du client et la clé secrète du client.
  • Les appels du service Web du portail sont authentifiés par les certificats de service.
  • Le service Web du WebAPI devrait avoir un certificat de serveur unique.
  • Le portail devrait avoir un certificat de client unique.
Application Web de Domaine commun de la FJGC

Considérations en matière de sécurité

  • Il devrait s'exécuter avec un compte de service du bassin d'application lequel a accès à la base de données de la FJGC.

Authentification

  • Aucune. Tous les accès à cette application Web sont anonymes.

Application Web des extensions des ADFS

Considérations en matière de sécurité

  • Il devrait s'exécuter avec un compte de service du bassin d'application lequel a accès à la base de données de la FJGC.
  • Il doit être en mesure de se connecter au service de mémoire cache sur toutes les applications frontales sur le Web de SharePoint.

Authentification

  • Aucune. Tous les accès à cette application Web sont anonymes.
Mémoire cache

Considérations en matière de sécurité

  • Il ne doit as être exposé à Internet.
  • Il a seulement besoin d'accepter les connexions du service Web des extensions des ADFS.

Authentification

  • Aucun
