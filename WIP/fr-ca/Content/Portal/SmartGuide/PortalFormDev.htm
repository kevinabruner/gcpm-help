Conception de formulaires pour le portail avec SmartGuide

Le logiciel SmartGuide Designer sert à concevoir des formulaires pour le portail. La plupart des documents dont vous pouvez avoir besoin pour SmartGuide sont inclus dans l'aide en ligne du GPSC : Documentation de SmartGuide d'Alphinat

Utilisez l'interface de protocole d'application (API) de SmartGuide pour étendre la fonctionnalité de SmartGuide. Certaines des extensions qui sont utilisées pour le portail du GPSC sont décrites ici.

SmartGuide Designer

Le logiciel SmartGuide Designer sert à créer des formulaires en glissant-déplaçant des zones sur des pages, en pouvant ensuite les subdiviser (les groupes sont répétables – cela permet une fonctionnalité semblable à celles de la représentation de grilles et de tableaux CROWE). Il y a deux types de zones sérialisées sur les formulaires : les zones de saisie et les zones de métadonnées. Elles se caractérisent par les critères suivants :

  • Zones de saisie : Zones dans lesquelles l'utilisateur inscrit des données que nous voulons saisir dans CRM et qui ont une zone d'entité jumelée. Lorsque le formulaire est sérialisé, ces zones sont regroupées dans le format XML, selon leur entité d'appartenance. La convention de nommage des zones qui représentent une zone d'entité de CRM ests entity_name.field_name dans le nom interne de la zone dans SmartGuide.

    Groupes de répétition : Zones qui font partie d'une relation N:1 qu'il faut saisir  (par exemple, les coûts d'un projet dans le cadre d'un projet) en utilisant cette convention de nommage : entity_name.field_name . Dans ce cas, I'entité ‘une’ suivie du nom de relation de cette entité avec l'entité Plusieurs : N.

  • Zones de métadonnées : Zones qui sont incluses dans le XML sérialisé, mais qui ne sont pas mises en correspondance avec une zone d'entité dans CRM. Une zone qui est considérée comme sérialisée, mais qui ne suit pas de convention de nommage de entity_name.field_name sera sérialisée en tant que nœud enfant du nœud de métadonnées. Ces nœuds peuvent être importants pour stocker des renseignements entre le portail et CRM afin de réduire les aller-retour de requêtes ou l'information qu'il faut enregistrer entre sessions de formulaire, mais qui ne sont mises en correspondance avec aucun registre dans CRM.

Les zones sont sérialisées en fonction de ce qui suit :

  1. Si elles sont marquées comme étant persistantes, elles sont automatiquement sérialisées. Par exemple, des zones cachées peut renfermer de l'information comme des identificateurs que nous voulons envoyer à CRM, ou des JSON pour lesquels nous avons exécuté une requête à partir de CRM au moment de lancer le premier formulaire.
  2. S'ils sont un type d'entrée (texte, date, sélection, etc.), ils sont sérialisés et groupés selon leur convention d'appellation.

Les champs Configuration ne sont pas sérialisés dans le XML. Ils sont utilisés pour stocké les renseignements dans la séance de formulaire actuelle et aident à fournir du contexte. Les éléments suivants sont communs et nécessaires pour tous les formulaires :

  • site : un champ caché par un paramètre de requête, c'est l'URL relatif a serveur Web de l'entité de zone actuelle. En d'autres mots, il dénote l'utilisateur, le projet ou le client actuel.
  • referenceNumber : un champ caché qui est aussi rempli par un paramètre de requête, il est un identifiant facile à utiliser pour l'instance de formulaire actuelle.
  • xml : un champ caché qui conserve le XML sérialisé, utilisé sur le téléchargement du XML sur le formulaire afin d'être désérialisé et le XML sérialisé après le traitement du formulaire pour enregistrer sur SharePoint (ou soumettre à CRM).
  • debug : un champ caché utilisé pour appuyer la pseudo connexion lorsque le formulaire a des problèmes, ce champ peut être basculé vers une zone de texte pour aider au débogage.
  • isSubmit : une valeur booléenne passé au service Web SaveDraft. Lorsque le bouton Soumettre est appuyé, il bascule de faux à vrai - dénotant que cet enregistrement est maintenant soumise.
  • userCanEdit : une valeur booléenne qui détermine si l'utilisateur a les permissions appropriées pour enregistrer / soumettre le formulaire. Si définit à faux, les boutons enregistrer et soumettre sont cachés.
  • SubmissionPDF : un champ caché qui contient la valeur codée base 64 des contenus PDF générés lorsque la soumission du formulaire est déclenché.
  • saveResult : Une valeur booléenne qui représente la valeur retourné lorsque qu'un enregistrement ou une soumission est réalisé. Important : ce champ est actuellement un champ de texte qui utilise CSS pour le cacher. Le raisonnement est qu'il a besoin d'avoir une classe CSS qui lui est attribué (élément d'enregistrement) afin de déclencher correctement un code dans le fichier Thème pour offrir une rétroaction à l'utilisateur (réussite ou échec). Ce champ doit aussi exister sur chaque page avec chaque copie qui a leur valeur par défaut défini à la valeur du champ d'origine (le champ d'origine est cartographié au résultat de l'appel au service Web saveDraft).
  • Fc_Profile_ID : sur les formulaires au niveau du projet, il est récupérer au téléchargement du formulaire et utilisé pour les requêtes au CRM.

Currency Fields : les champs qui ont pour but de refléter les valeurs de devise ont besoin d'avoir une configuration spéciale afin d'appuyer la concrétisation adéquate du caractère « $ » pour l'affichage actuel. Trois pièces principales de configuration ont besoin d'être utilisées afin de l'appuyer :

  • Avoir le préfixe du champ de la devise dans les champs de l'onglet « Apparence » assigné à « $ ».

  • Avoir un classe CSS « amountnumeric » assignée dans l'onglet « Apparence »

  • Dans le document de traduction, avoir le « $ » en suffixe pour la colonne Français

Pièce jointe du document

Chaque page a un bouton Joindre / Voir documents dans le cadre du thème WET_THEME. Ce bouton ouvre une boîte de dialogue qui permet aux utilisateurs de télécharger des documents (en naviguant ou en utilisant la fonction de glisser-déposer) au dossier de soumission du formulaire pour l'ébauche actuelle du formulaire.

Structure du Smartlet du GPSC

Les formulaires du SamrtGuide (Smartlets) suivent une structure générale :

  • Une ou de multiples pagesà pour recueillir l'entrée par l'utilisateur. Mis à part le contenu pour recueillir l'entrée de l'utilisateur, chaque page a aussi des boutons pour :
    • Enregistrer - un champ de la configuration de l'enregistrement pour conserver le résultat d'une opération SaveDraft
    • Navigation Précédant et Suivant
    • Joindre /  Voir les documents.
  • Page d'examen

     La page d'examen est composé de l'outil de récapitulation des données de SmartGuide, un bouton Soumettre, un champ de la configuration du résultat de la soumission pour conserver les résultats de la soumissionSaveDraft, un bouton Joindre / Voir les documents et un bouton Précédent. La récapitulation de données permet au concepteur du formulaire d'inclure et d'exclure les données qui n'ont pas besoin d'être reflétées à l'utilisateur dans le cadre de leur soumission. La génération de PDF est basée sur cette page ou sur l'outil de récapitulation de données. Après avoir cliqué sur soumettre et avoir généré avoir succès l'aperçu PDF et envoyé les données appropriées au CRM, le formulaire avance à la page Réussite

  • Page Réussite

    Elle informe un utilisateur que le formulaire a été soumis et affiche un numéro de référence. Les utilisateurs peuvent accéder au bouton Joindre / Voir les documents, mais ne peuvent pas faire de changement aux documents de soumission (XML et aperçu PDF) et tous les fichiers correspondants sont en lecture seule.

Connexions

Deux types de connexions sont configurées - les fonctions d'extension d'API et les appels de services Web de SharePoint.

Fonctions d'extension
Fonctions d'extension

Fonction d'extension

Description

SmartletSerializer

Cette fonction se produit lors de l'enregistrement ou la soumission du formulaire.

Il évolue par le formulaire et sérialise les champs d'entrée et les métadonnées.

Le nœud racine est nommé d'après le nom interne du Smartlet avec ses nœuds enfants qui sont des métadonnées et toutes les entités qui existent sur le formulaire.

Les Métadonnées enfants sont les champs qui sont indiqué comme étant possible de sérialiser, mais qui ne suivent pas la convention d’appellation entity.field.

Les nœuds enfants ont leurs champs comme enfants directs. Le fil XML sérialisé est conservé dans le champ de configuration nommé « XML»

GeneratePDF

Cette fonction se produit à la Soumission du formulaire.

Il génère le fil codé based64 de la représentation PDF du formulaire et de l'entrée de l'utilisateur.

Les contenus du PDF sont déterminés par ce qui se trouve dans le champ de récapitulation des données sur la page Examen.

Pour les changer, dans la fonction d'extension GeneratePDF, vous pouvez changer quelles pages sont converties dans l'aperçu du  PDF en utilisant la convention suivante lors de la création du tableau d'octets :

byte[] bytes = smartlet.generateFile(Constants.FileType.PDF, $"P{page1Id}}:PrintOut |P{page2Id}:PrintOut");

Essentiellement, la convention d’appellation est ‘P’ + IdOfPageToAdd + ‘:’ + ‘NameOfTemplate’.

Toutes les pages additionnelles suivent la même convention d'appellation avec un caractère de pipeline « | » entre le nom du modèle et le caractère « P ».

InitializeRequestParameters

Cette fonction tire les données des paramètres de requête et défini leurs champs de configuration correspondants.

Il se produit à l'initialisation du Smartlet.

InitSmartlet

Cette fonction se produit au chargement du formulaire, il appelle le service WebLoadDraft webservice and SmartletDeserializer.

Les autres appels à la fonction d'extension et au service Web peuvent être fait de façon à être plus dépendant du formulaire.

ImportCsv

Cette fonction permet à l'utilisateur de télécharger un modèle offert, le remplier avec les données et le télécharger sur le formulaire afin de créer des rangées dans un groupe répétitif.

Deux paramètres, le nom du champ de téléchargement du SmartGuide et le groupe répétitif ont besoin d'être fourni.

Important : l'utilisateur doit téléverser un CSV avec les entêtes qui correspondent aux étiquettes des champs du groupe répétitif. Le formulaire ne prend pas en compte si la langue du modèle correspond à la langue actuelle de SmartGuide (peut télécharger le modèle Anglais dans le contexte du formulaire Français et vice versa).

Par contre, il suppose que l'entrée correspond à la langue CSV - ce qui signifie que les devises suivront les formats de devise en français dans le modèle Français s'ils utilisent un formatage de la devise. Les groupes répétitifs qui ont des champs déroulant ou à sélection ou les champs calculés seront aussi ignoré et ne seront pas nécessaire dans le modèle CSV.
Connexions SharePoint

Elles contiennent tous les appels au service Web que le formulaire réalise. Tous les appels de service Web à CRM sont effectués par le service Web SharePoint.

Pour de plus amples renseignements, consultez Service Web SharePoint

  • Lors de l'ajout de nouveaux services Web, il est important de comprendre les entêtes suivantes ainsi que leurs valeurs par défaut.
  • Lors de l'ajout de l'URL au service Web, un paramètre additionnel {Client-Site} devrait être pris en charge au préalable à l'emplacement où le service Web est conservé.
  • Lors de la création de la référence au service sur le Smartlet, il devrait être défini à la valeur de la variable de la configuration « site ».
