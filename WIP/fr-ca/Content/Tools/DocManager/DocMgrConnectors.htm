Connecteurs du gestionnaire de documents

Les connecteurs sont responsables de l'authentification de l'utilisateur par rapport Système de gestion des documents (SGD) externe et de faire des appels à ce dernier.

Le plugiciel du gestionnaire de document détermine le connecteur du SGD à utiliser uniquement pour l'option « Dépôt de documents » établie qui se trouve dans l'entité CFG d'entité du document, au moyen de la valeur d'option afin d'appeler de façon dynamique le bon connecteur du SGD.

Prêtes a l'emploi, ces options sont cartographiées comme suit :

Par exemple, si vous sélectionnez l'option SharePoint, le plugiciel appellera l'action personnalisée egcs_dm_Execute1 and use the egcs_Username1, qui sont cartographiés au plugiciel connecteur SharePoint et qui sont pris par ce dernier.

Remarques :

  • Si vous changez les mappages, assurez-vous que l'action personnalisée correspondante est associée au bon plugiciel connecteur.
  • Si vous prolongez le plugiciel afin de soutenir d'autres SGD externes, ajoutez une nouvelle option à l'ensemble d'options de dépôt de document et créez l'action personnalisée correspondante et le champ du nom d'utilisateur  (dans l'entité d'utilisateur du système) avec la même convention d'appellation et les mêmes paramètres.
SharePoint

REST API de SharePoint est utilisé pour faire des appels à SharePoint. Les appels sont authentifiés en envoyant le nom d'utilisateur et le mot de passe du compte de service à SharePoint afin de récupérer le jeton SAML, le témoin FedAuth, et le bulletin de formulaire, qui ne sont pas cachés.

Parce que le plugiciel gestionnaire de documents est une application externes à SharePoint qui utilise l'API du côté client, il ne peut pas authentifier ou imiter l'utilisateur appelant sans le nom d'utilisateur et le mot de passe de l'utilisateur, donc l'utilisation d'un compte de service d'administrateur est requise. Les champs CreatedBy and ModifiedBy seront par conséquent étiquetés avec le compte de service.

Les développeurs peuvent surclasser le mécanisme d'authentification SharePoint du plugiciel, afin de mettre en œuvre localement la mise en cache, l'imitation de l'utilisateur ou l'authentification directe de l'utilisateur. Le connecteur SharePoint peut également être surclassé si les développeurs choisissent d'utiliser des API de rechange a SharePoint, comme le Client Side Model Object (CSOM) de SharePoint.

Pour obtenir de plus amples renseignements, consultez API du plugiciel de gestion de document.

GCDocs

GCDocs utilise OpenText comme système de gestion des documents, lequel offre un point terminal de service Web fondé sur SOAP pour l'utilisation d'applications externes. Le plugiciel authentifie à GCDocs au moyen du nom d'utilisateur et du mot de passe fourni d'un compte de service administrateur et procède à imiter l'utilisateur appelant pour tous les appels subséquents à GCDocs.

Certains appels sont effectués sous le compte de service afin de récupérer les données auxquelles l'utilisateur n'aurait peut-être pas accès; toutefois, les données récupérées de ces appels ne sont jamais retournées à l'utilisateur et tout appel modifiant GCDocs sont effectués sous l'utilisateur appelant. De cette façon, les permissions d'utilisateur sont respectées et les utilisateurs ne puissent pas faire les actions que l'utilisateur appelant ne serait normalement pas en mesure de faire dans GCDocs.

Il y a seulement une façon de s'authentifier et de se connecter à GCDocs de façon programmatique, donc aucun surclassement n'est fourni.
