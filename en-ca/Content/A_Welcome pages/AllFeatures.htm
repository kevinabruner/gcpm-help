All GCPM features at a glance

A short description of each key feature in GCPM. Follow the "See also" for all documentation related to the features below.

Stakeholder features

Account - in brief

Organizations, whether businesses, government departments, or not-for-profits, are managed with accounts, and individuals are managed with contacts. The account is a native CRM Dynamics feature leveraged in GCPM to enable a client-centric approach to managing and reporting on grants and contributions.

Key account features include: 

  • Track activities* (emails, phone calls, meetings, etc.) that your organization has had with each stakeholder or client. A built-in view of "My Activities" acts as your daily “To Do” list. Activities ensures interactions with customers can be found later and reassigned if needed.
  • Segment accounts into marketing lists*, to communicate with targeted audiences.
  • Create search queries, personal views*, and reports* to track activities by customer or activity type.
  • Use dashboards and charts* and Import and export data*.

Accounts can be accessed across your organization, with more sensitive data related to project delivery available only to delivery staff on the GC Client record. Dynamic relationships between the account and other records provides, at a glance, project data and stakeholder interactions, such as:

  • Communications work done for prior funding announcements
  • Claims or receivables from current or prior projects

Accounts can have multiple, flexible relationships to account-related data:

  • Contacts (individuals) are linked to accounts with dynamic roles to better manage engagement and communication with stakeholders. For example, a consultant working with multiple companies can have those connections reflected in GCPM.
  • Hierarchical relationships: parent / subsidiary relationships between companies, or departments within post-secondary institutions
  • Roles can capture collaborations between non-profit partners

GC Client - in brief

There is one GC Client record for each account that has applied for funding, has active project(s), is a co-obligant on a project, or grants and contributions business with the agency. The GC client record displays a summary of an account’s application and funding history, and tracks G&C-related engagement activities.

GC client records enable these client-centric services: 

  • Managing a client's funding agreements as a portfolio
  • Conducting client risk evaluations
  • Recording client financial statements and reviews
    For example, a client who has more than one active project (e.g. new application, plus active project in repayment or control period) needs only submit financial statements once. The statements are available on the client record for internal management purposes.
  • Updating and maintaining monitoring for one or multiple projects, which may be at different stages.

GC ClientA child record of an account that has a funding relationship with the RDA. It stores all grants and contributions management info related to the account. Any organization that submits applications or receives funding must have a GC Client record. records have more limitations than an account record. For example, hierarchical relationships that are possible with accounts are not possible with GC Clients. For example, a university may have multiple departments, which can be captured in GCPM as child accounts of the university. Separate departments may apply for funding, and manage a project, but as the university is the legal entity that signs the contribution agreement, only the university account has a GC Client record.

Program features

Amendment feature - in brief

Amendments is a GCPM tool to safeguard the integrity of contribution agreements (CAs) over time. The amendment feature locks / unlocks GCPM fields that appear in a CA after an assessment has been approved. Amendments can be made during the CA negotiation stage, as well as the project administration stage. Amendments are used for formal amendments as well as administrative / internal changes related to the project that require modifying the fields displayed in the CA.

GCPM includes a number of pre-configured amendment types (e.g. administrative amendment, substantive 2-way amendment, etc. as well as default). For each type of amendment, a specific set of transitions, validations and approval path has been designed. Data integrity of the project is preserved: an amendment does not overwrite data until approval has been granted for the changes.

Amendments are captured with a snapshot; these are available in the amendment grid.

Application - in brief

The application feature includes the application form, where questions (unique to a program, stream, or intake) are configured. The application entity does not support displaying information in a grid (e.g. a project cost table), but clients are able to submit this data in a separate document provided online. The document and application are captured in a PDF and is available for assessment.

The application status is a configurable lookup, and can be built into business processes to trigger communication or other activities. The internal status of an application can be mapped to an external status for clients.

For example, a client could see "application received" and the internal status could be one of these: "received / under review / converted to project"

Clause feature - in brief

The clause feature provides flexible tools to manage project-specific clauses for contribution agreements. A clause template for a program contains program-specific clauses that are copied to each project in that program. Officers may be able to edit, modify or remove the clauses that are inserted to the CA for the project , depending on how the clauses are configured. One clause template can be used in multiple programs or streams.

Special Conditions are project-specific clauses. These may be standard clauses, editable by an officer, or new clauses manually added by an officer.

GCPM includes one example of a CA document with two unique clause templates under different programs (WES – Entrepreneurship Fund and the GCPM Core Program). In the sample GCPM CA, special clauses are inserted in the Special Conditions section.

Disbursement - in brief

GCPM disbursement offers full management of processing project funds for clients. Clients submit claims for costs incurred, and advance requests when they need funds in advance of incurring costs. Each claim can be reviewed separately, or bundled for consolidated review. A disbursement review is conducted by officers to ensure validity of claims and advance requests. The review can handle one, or multiple, claims and advance requests. Disbursement reviewA disbursement review can include one or more claim requests and advance requests. Disbursement reviews are the best! is integrated with business process flow, decisions and automated actions to ensure your organization's review and decision-making processes can be consistently applied across projects and clients.

Clients can: 

  • Submit claims and advance requests via the portal, when it has been configured by your organization.
  • Submit claims manually (by email, mail or other means). Officers can manually enter claims into GCPM.
  • View records they have submitted to the RDA. These records are preserved as submitted. Any review, change, or adjustments are reflected in the associated disbursement review.

GCPM provides the ability to capture specific invoice and cost related information at the claim level, and similar information for an advance request. RDA’s can add different fields to these functions to customize it to the level required by their claim review / payment process.

In a disbursement review, officers can:

  • Approve, delete, defer, adjust, or split cost items to treat one claim with several actions.
    For example, a client submits a large invoice. Upon review, it is determined that only part of the expense has been incurred to date. The officer can split the invoice, and recommend the incurred items are paid, and defer the balance to a later time. Deferred items can be included in future disbursement reviews, without the client having to resubmit.
  • Manage hold back
  • Implement payments or recalculations as the result of amendment to the funding formula on the project
  • Trigger repayments or receivables required during the active phase of a contribution.
  • Code claim items to specific fiscal years
  • Indicate if the payment amount should be set up as a PAYE
  • Reconcile all or partial amounts of outstanding advances
  • Issue new advance amounts.
  • Create negative payments when over-payments or un-reconciled advances need to be clawed back
  • Complete a digital checklist to ensure due diligence on claim review. This disbursement checklist can be tailored to each program, and captures audit trail information data required for claim processing.

Disbursement conditions - in brief

Disbursement conditions are associated with the project profile. Officers can review disbursement conditions to determine if the project is in a disbursable state.

For example, an ongoing disbursement condition requires the monitoring of other project funding sources. An officer determines there is inadequate evidence of matching funds, and changes the status to "not disbursable." When an officer attempts to seek s.34 approval for a disbursement review, GCPM prevents payment until the issue is addressed.

Note: the use of the disbursable state in business process transitions is configurable to your organization's requirements .

Financing feature - in brief

The financing feature...includes cost items, cost categories and subcategories...

A project financing grid displays sources of financing (excluding departmental funding) that will pay for the costs of the project. Each financing source can have these attributes:

  • Category / subcateogry
  • Confirmation (yes / no)
  • Stacking % (percent of the financing source that should be considered for stacking calculation.

The financing balance is a calculated field. Formula: Financing balance = Total costs - total departmental amount.

Priority feature - in brief

The priority feature can be used to prioritize applications based on responses to application questions, or prioritize projects based on responses to assessment questions. It is quite similar to the assessment feature.

The WES program demonstrates a basic priority assessment based on how the client responds to the questions, which are scored in the WES application.

Project assessment features - in brief

A number of GCPM tools record due diligence and assessment of projects before approval. Any data captured during assessment (responses to assessment questions, costs table, budget, financing, eligibility, etc.) can be captured in documents, such as the contribution agreement, letters to client, etc.

Assessments include: 

  • Assessment questions
  • Cost / budget
  • Disbursement conditions
  • Financing
  • Performance indicators and targets
  • Eligibility
  • Priority
  • Risk

Transitions - in brief

Transitions is a configuration tool, used in conjunction with business process flows and actions, to manage the life cycle of business records (e.g. project life cycle, disbursement review, etc). Your organization can use transitions to configure when to: record decisions, capture snapshots of data (at a moment in time) , lock records, and many other automated actions.

Risk assessments can be generated as part of the project life cycle, (e.g. integrated into the assessment process) without requiring officers to navigate and undertake risk as a separate activity.

The GCPM XX (name) includes transitions related the negotiation and execution of a CA.

Bilingual plugin - in brief

The bilingual configuration tool (CFG Bilingual) is used to extend translation options beyond that provided by Dynamics tools (see notes, below). The GCPM bilingual configuration tool configures these types of data to display in English or French, according to a user's language of choice:

  • Lookup values
    e.g. clauses for contribution agreements are available in both English and French. Depending on the language of the contribution agreement, clauses are inserted in the appropriate language.
  • Entity names
    e.g. Disbursement review / Examen de débours)
  • Field names
    e.g. Operating name (EN) / Operating name (FR)
  • Entity "name" field (see Configure account names for further detail)

  • Filter records within the grid

Document generator - in brief

The document generator can generate any document your organization may need to manage clients and projects. Business users can modify document templates to display CRM data. The document generator is a tool that takes snapshots of data and converts it to human-readable format (e.g. a document, a report or web page). The document generator provides flexible formatting and content design, and can readily tailor documents to reflect the look and feel of your organization.

Document templates can be configured to display project-specific information. Conditional text, such as clauses determine by assessment outcomes and project data, is configured using XML code that is linked to GCPM fields.

A configurator defines the data fields to include in a snapshot, and the snapshot populates an MS Word document template with data in XML format.

GCPM includes a configured CA template.

Financial integration

Financial integration (FI) describes the components of GCPM that manage the relationship between GCPM and the financial system. This includes all coding required to set up accounts for direct deposit or automatic withdrawal for repayments, and ensures the structure of vendors and customer records are respected in the financial system without requiring multiple GCPM accounts to for separate bank accounts.

Portal - in brief

The portal is configured to display application forms designed in GCPM to portal users. Portal application forms can be

  • Static: a common set of fields and questions is displayed. In this case, each program / initiative may have a separate application form.
  • Interactive - as users enter information about their project and organization, the data presented to the user dynamically changes. Questions can be tailored based on input from the client

Portal accounts will interact with the GCPM account records. The portal will facilitate the creation of new accounts, and the management of client information, flagging where necessary when requested changes / updates are required.
