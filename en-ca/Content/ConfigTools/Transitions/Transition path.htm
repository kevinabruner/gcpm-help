Configure a transition path

Transition paths identify the possible routes that records can travel through a business process flow.

To configure a transition path: 

  1. In a transition template, add a new record to the Transition paths grid.
    The transition path form opens.

  2. Complete the fields using the onscreen help and the notes below: 

    • Name: Type a name for the transition path (EN, FR).
      The name of a transition is displayed to the end user in the transition dialogue, so it is important to provide a meaningful name that has intuitive meaning to them. e.g. this could include the status it is transitioning to, or action(s) that occur in the transition.
      Example: "Assessment complete"
    • Transition template: this field is auto-populated when creating a path within the template grid.
      Displays the template associated with this path.
    • Business process flow: this field is auto-populated.
      Displays the BPF associated with this transition status. This field acts to filter the options available in the active and target stages, below.
    • Order: Provide a number, used to sort and identify this transition path.
      Example: If the status "assessment complete" is associated with the number 100, enter 100 in this field.
  3. Status and stage transition
    • From project status: Choose the initial status a record must have to display this transition path.
    • To project status: Choose the target status for the record once this transition path is completed.
    • Active stage: Choose the stage the record must be in to display this transition path.
    • Target stage: Choose the stage for record once this transition path is completed.
    • Target form: This is an optional field. Leave this field blank if no form change is required. If a form change is required, select the appropriate form to display after the transition.

  • For example, start the process using the EGCS BASE form and switch to the PAF form after the transition.

Decision / Approval

  1. Decision type: Choose a decision type that will be created and stored in Decision History after the decision is made.
  2. Decision comments: What is this for...
  3. Approval type: Optional - If an approval is required on this transition path, choose an approval type. This option defines which roles are required to provide approval for this path.
    For example, Manager, director, director-general may be defined for an approval type.
  4. If an approval type is selected, three additional action sets are displayed. Any of these can be provided (optional):
    1. Pre-approval (action set):  A set of actions that are called before sending to the desired status. For example, it may lock a record that is sent to an approver, or verify that certain fields have been filled in (e.g. date fields).
    2. Notification (action set): Creates an Approval task and assigns it to the next approver. Once the next approver makes a decision, it will mark the activity as completed.
    3. Deny (action set): A set of actions that is triggered if the approver denies the recommendation. For example, the record may be unlocked and return to the officer for further work.
  5. Transition (action set):  Choose a set of actions to run before moving to the target status. If an approval has been configured for this path, this action set will run before / during / after?the pre-approval / notification or deny action sets.

Examples

  • A transition action set can validate that certain fields have been filled in, such as project description, before moving to the next status.
  • If a validation fails (for example, the project description is blank), the user will remain at the same status until a description is provided.
