Naming conventions

Solution Publisher Naming

Publisher prefix for the Best Base solution will be "egcs" and "gcpm" for the GCPM implementation. This document will be updated as new standards are required.

Entity Naming

There are three names that are mandatory when creating entities in Dynamics CRM:

  1. Display Name
  2. Plural Name
  3. Name

Display Name and Plural Name should describe entities and no naming convention will be enforced on these values.

However, for the Name (the internal name) field the following rules apply:

  1. The name of all the entities for the Best Base and GCPM solutions will be in the format "egcs_EntityName" and "gcpm_EntityName" respectively
  2. Use Camel casing for delimiting words or abbreviations in the entity and field names. Camel case: Capital letter starts each word/token as in the humps on a camel. e.g. ThisLongEntityName
  3. The first two characters of an entity name will be capital letters, depending on the functional group to which they belong:
Function group Two character prefix
Stakeholder function ​ **SH**
G&C client function​ **​SG**
​Funding case function ​**FC**
​ Program functions (transfer payment) ​ **​TP**
Planning and performance measurement functions **​PP**
​Agency wide functions **​AY**
Attributes (Fields) Naming

Entity fields will follow these naming conventions:

  1. Name of all fields for the Best Base and GCPM solutions will be in the format "egcs_FieldName" and "gcpm_FieldName" respectively.
  2. The first several characters of the field name will represent the field type for these types only (ignore for other types)
    insert table

  3. Lookup names should contain the prefix only once:

    gcpm_EntityName instead of default gcpm_gcpm_EntityName

    Display name: Related EntityName

  4. Fields used for storing the values of the English and French translations of the entity name will have the suffice EN or FR (gcpm_Name_EN, gcpm_Name_FR)

    • Every entity's name will have these additional fields
    • Fields whose data is human readable such as a "status" field ("Approved" / "Approve")

    Examples: gcpm_LegalName, gcpm_YesNoSame_As_Mailing_Address, gcpm_DateStart, gcpm_AmtRepayment

Relationship Naming

Relationship names should contain the publisher prefix once at the beginning. The default name provided by CRM will include the publisher prefix multiple times so pay careful attention.

The basic rule is to concatentate the publisher prefix with the primary entity name and the related entity throug the use of underscores:

[Publisher prefix][Primary entity][Related entity]_[Optional additional Custom Name]

Example 1: A relationship between SG_Client and FC_Client_Risk

  • Name = gcpm_SG_Client_FC_Client_Risk

Example 2: Two relationships between Contact and FC_Profile for different purposes, thus requiring the additional naming at the end to distinguish (noted in bold):

  • Name: gcpm_Contact_FC_Profile_Primary_Contact
  • ​​Name: gcpm_Contact_FC_Profile_Signatory
Intersect Entities

An intersect entity is an entity with a 1:N relationship with entity X and an N:1 relationship with entity Y. The result is two entities with an effective N:N relationship with one another.

  1. Intersect entities will be named [Publisher Prefix][Primary Entity 2 character prefix][Primary entity name with underscores removed][Related Entity with underscores removed] eg. intersect entity that links SG_Client with FC_Client_Risk

    Name = gcpm_SG_SGClientFCClientRisk

  2. A relationship with an intersect entity follows standard naming convention:

    gcpm_SG_Client_SG_SGClientFCClientRisk

Forms Naming
  • For each entity create a new main form named "EGCSBase Form" or GCPM Base Form"
  • Each entity will have a base form with the Description = "Base form for EGCS" or "Base form for GCPM"
  • Each base form will have all custom fields in alogical order as determined by the program specialist.
  • For each relationship with 1:N with the entity, add a tab with a sub-grid. Name the tab with the name of the entity it is related to
  • Sub-grids should be minimized by default
Form Elements Naming

It is important for the internal names of form objects to follow a standard naming because JavaScript (and possibly other things) will be clearer to read and write. The following are proposed naming conventions:

Form element Prefix
Tab tab_
Section sec_
Sub-grid grid_
Quick view qv_
Timer timer_
