Filtering a Lookup

 

Note:  Some of the functionality outlined in this section can also be implemented using the Advanced Lookup Options on the column configuration screen.

 

 

EditAble CRM Grid exposes API commands to support filtering lookup controls when editing and inserting a row in inline edit mode.  You can filter your lookup results using nearly any field of the entity, not just other lookup fields.  For example, you could filter the lookup by StatusCode or a text value.

 

Generally, there are two main reasons to filter a lookup control:

1.To filter the options available on the lookup window so they are more relevant to what you want the user select.  For example, when selecting an Account you may only want a certain type of Account shown.

Note: This can also be done by using a different CRM View as the basis for the lookup.  That is outlined in the section Changing the Lookup View.
 

2.To filter the options available on the lookup window based on another field that the user has populated.  This is referred to as a dependent lookup.

          

 

To show how to implement filtered lookups, we are going to use an EditAble CRM Grid based on the Account entity.  For each Account record with a Relationship Type of "Prospect", we want to allow the user to select the Account and Contact that referred them.  We have created two new Lookup fields in the Account entity that we’ll add to our EditAble CRM Grid: new_ReferringAccountId (a Lookup to the Account entity), and new_ReferringContactId (a Lookup to the Contact entity).

 

Here is the functionality we need to implement using EditAble CRM Grid’s filtered lookups:

1.The Referring Account is always going to have a Relationship Type of “Influencer”, so we only want the lookup to show Account records of that type.

To accomplish this, we’ll need to add logic to the On Edit and On New events that sets the static filter for the Referring Account field.
 

2.The Referring Contact lookup should only show contacts from the selected Referring Account (i.e. where the Contact’s Parent Customer is the Referring Account).

To accomplish this, we’ll need to add logic to the On Change event of the Referring Account field.  Additionally, we’ll want to add logic to the On Edit event that will filter Referring Contact for records where Referring Account has already been populated.

 

 

In the sample code, we have set up 3 functions that we wind bind to EditAble CRM Grid events:

 

·GridExamples.Accounts.OnEditEvent – This gets attached to our EditAble CRM Grid’s “On Edit” event.  This function invokes sub-functions that will change the view that our Referring Account lookup uses, apply a filter to our Referring Account lookup, and apply a filter to our Referring Contact lookup.

          

·GridExamples.Accounts.OnInsertEvent – This gets attached to our EditAble CRM Grid’s “On New” event.  This function does everything the OnEditEvent function does, except that it does not have to loop through multiple lines and it uses the NewRowIndex constant, because new rows do not have a RowIndex applied to them.

          

·GridExamples.Account.OnChange_ReferringAccountId – This gets attached to the “On Change” event for the new_ReferringAccountId column.  This function will invoke a sub-function that filter the Referred Contact lookup.

          

 

We’ll look at each sub-function in detail, since that is where the main logic is taking place:

 

FilterReferringAccount

This is the function we will call to apply a base filter to the Referring Account lookup. 

 

 

 First, we are creating our filter expression, using GetFilterExpression.  There are 3 parameters we need to send:  The name of the field (in the parent entity) we want to filter on, the operator, and the value.  In SQL terms this line is basically creating a WHERE clause of:  Account.CustomerTypeCode = 6.

 

You’ll notice we didn't actually specify "Account" in the filter expression.  That is because EditAble CRM Grid already knows about the entity each Lookup references, so when we apply this filter expression to the new_ReferringAccountId column (next line of code above), it knows the filter is based on the Account entity because that is what the new_ReferringAccountId field is based on.

 

As mentioned, the next line of code simply applies the filter expression to the column we want to filter.  The rowIndex that was passed into the function was originally retrieved from the either the "context" parameter of the On Edit event, or the NewRowIndex constant for the On New event.

 

 

FilterReferringContactByReferringAccount

This is the function we will call to apply a filter to the Referring Contact lookup.

 

 

This function is similar to the one above, however we are going to filter the Referring Contact lookup to a dynamic value (the Referring Account) instead of a static value.  For that, we simply use GetFieldValue to return the current value of the Referring Account field.  This is what we’ll send to the GetFilterExpression function.

The other thing we are doing here is changing the state of the Referring Contact control.  By filtering Referring Contact by Referring Account, we are basically saying that it is dependent on Referring Account being populated.  For this, we just check if Referring Account is empty or not.  If it is empty, we don’t want users moving ahead a picking the Referring Contact first, so we just disable it until the Account is selected.

 

 

Note:  Filtering lookups is only supported for EditAble CRM Grid's default Lookup control.  The functionality will have no effect on Lookup columns that have been configured to use a picklist control.
