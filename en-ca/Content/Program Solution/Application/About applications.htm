About applications

Clients can submit applications for funding in two ways: online, or manually (by email, mail, fax, etc.). When a client clicks "Submit:"

  • Data provided by the client is transferred to the appropriate fields in a GCPM application.
  • A snapshot of the data is captured in a non-editable PDF. The PDF is stored in a location accessible to the client as well as staff at your organization.

An application must be configured in two places: first in GCPM, and on the portal. The portal form fields must be mapped to the GCPM application form fields; and web services and other settings are configured to transfer online data to the appropriate GCPM application.

Automated priority assessment may be configured to score and rank applications based on applicant responses. Priority assessments facilitate workload management, esepecially for infrastructure and other high volume programs.

The WES program provides an example of a priority assessment on applications.

Configuring applications on the portal

When a client "Submits" an online application, data is transferred to GCPM using web services. There are several configuration tasks related to the portal: 

  • Design portal application forms using the SmartGuide Designer software. Portal fields are mapped to GCPM fields.
  • Create static or dynamic portal forms, and other options.
  • Modify or build your own web services to transfer data from the portal to GCPM.
Automation

  • A GCPM application is auto-populated with data received from portal applications according to what has been configured for the portal at your organization.

    Your organization should consider how to manage applications submitted by email, fax or other manual options.

  • Applications can be automatically scored using the eligibility feature.
Design considerations
  • The application is designed to map information from a client-facing form on a portal to GCPM, where it is converted into a project. Once it is a project record, an officer begins assessment. Intakes can be dynamically administered. Large intakes can automate the identification of duplicate and invalid applications to streamline and prioritize the intake process.
  • The examples included with GCPM provide examples of ways to incorporate dynamic and configurable questions between GCPM and a portal, as well as hard-coded information that can be mapped on a field by field basis. These examples help inform design decisions for local application processes and intakes.
  • Applications are not automatically registered as projects associated with accounts and GC clients. This was purposefully not developed as a variety of application and intake processes may be used by your organization. GCPM provides an example of a simple workflow (name) that can be run on demand for applications (either individually or in bulk), that creates accounts, GC clients, and project numbers from the application.
  • Your organization will need to carefully consider procedures to accommodate changes to account / GC client information, if they submit applications that contain different information than your existing records. Be careful, as changes to certain data fields may impact legal agreements already in place.
Backlog / future design
  • For the MVP, minimal auto-population of client information has been developed. Your organization must determine what information is on an application form, how to bring this into CRM.
  • Automatic conversion of applications to projects is not included in MVP. Your organization must decide how to manage incomplete applications, duplicate submissions, and applications submitted in error or withdrawn. Manual administrative action is required to associate an application to a project, and to a GC Client (or create new records, if applicant is new).

Associated tools
Application - in brief

The application feature includes the application form, where questions (unique to a program, stream, or intake) are configured. The application entity does not support displaying information in a grid (e.g. a project cost table), but clients are able to submit this data in a separate document provided online. The document and application are captured in a PDF and is available for assessment.

The application status is a configurable lookup, and can be built into business processes to trigger communication or other activities. The internal status of an application can be mapped to an external status for clients.

For example, a client could see "application received" and the internal status could be one of these: "received / under review / converted to project"

Portal - in brief

The portal is configured to display application forms designed in GCPM to portal users. Portal application forms can be

  • Static: a common set of fields and questions is displayed. In this case, each program / initiative may have a separate application form.
  • Interactive - as users enter information about their project and organization, the data presented to the user dynamically changes. Questions can be tailored based on input from the client

Portal accounts will interact with the GCPM account records. The portal will facilitate the creation of new accounts, and the management of client information, flagging where necessary when requested changes / updates are required.

Priority feature - in brief

The priority feature can be used to prioritize applications based on responses to application questions, or prioritize projects based on responses to assessment questions. It is quite similar to the assessment feature.

The WES program demonstrates a basic priority assessment based on how the client responds to the questions, which are scored in the WES application.
