About programs

Program - in brief

Programs are central to configuring many features in GCPM: 

  • Key program data to ensure timely and accurate reporting to senior management.
  • Templates to manage program-specific requirements for:
    • Applications
    • Assessment
    • Project lifecycle management - business processes that track project status, decisions
    • Amendments
    • Disbursement

RDA’s can configure unique templates for these features per program, or reuse the same templates across multiple programs.

At least one program and one funding opportunity must be configured to receive applications, to create project assessments, to develop and manage contribution agreements, to make disbursements and monitor projects. Your organization is likely to create a hierarchy of programs, with multiple intakes to manage the range of funding available.

The sample data included in GCPM illustrates features that your organization may review. Two programs are included:

  • GCPM simple program
  • Women Entrepreneurship Strategy, designed with a more complex program hierarchy and intakes.
    The program hierarchy permits reporting on projects at different levels of granularity, when a program terms and conditions has sub-programs, and even multiple streams. Each sub-program / stream can have multiple intakes that may accommodate unique applications, or timed intakes.

Programs are mainly configured by business "super users" and are not visible to officers. GCPM exposes only the relevant tools to users.

Key data captured in a program record
Key data captured in a program record
Category Details
Reporting data
  • Tombstone data for programs, e.g. name, program hierarchy, key dates, and the purpose of the program.
  • Facilitates reports such as proactive disclosure
Configuration data
  • Identifies current and historical templates used to manage projects and associated activities for the program.
  • Several technical tools use template data to create and / or manage projects, disbursements, amendments, and documents.

Funding opportunity
  • A program cannot have project intake unless the program is associated with a funding opportunity
  • The funding opportunity facilitates multiple intakes under one program, and can apply different assessment criteria to each intake
Financial data
  • Key financial information for reporting purposes. May be input manually, or through connection with the financial system.
  • GCPM MVP is intended to rely on financial systems financial information. Additional budget / fiscal year information displayed on the program profile is manual and serves for reporting purposes only
  • Financial coding tools to integrate with a financial system
Automation

Program configuration through the use of the clone plugin or action set configuration as part of transitions facilitates the automatic creation or copying of information in multiple features attached to project management.

Design considerations
  • Funding opportunities are required for any programs that have projects associated with them. The funding opportunity triggers the application of most assessment tools. Programs without a funding opportunity cannot have projects created.
  • Programs in a hierarchy have a parent-child relationship. However GCPM provides the option for a child program to identify the ultimate parent program (e.g. grandparent? ) as a “real program” to report without having complicated roll-ups through “middle” program records.

Associated Tools

Transitions - in brief

Transitions is a configuration tool, used in conjunction with business process flows and actions, to manage the life cycle of business records (e.g. project life cycle, disbursement review, etc). Your organization can use transitions to configure when to: record decisions, capture snapshots of data (at a moment in time) , lock records, and many other automated actions.

Risk assessments can be generated as part of the project life cycle, (e.g. integrated into the assessment process) without requiring officers to navigate and undertake risk as a separate activity.

The GCPM XX (name) includes transitions related the negotiation and execution of a CA.

Project assessment features - in brief

A number of GCPM tools record due diligence and assessment of projects before approval. Any data captured during assessment (responses to assessment questions, costs table, budget, financing, eligibility, etc.) can be captured in documents, such as the contribution agreement, letters to client, etc.

Assessments include: 

  • Assessment questions
  • Cost / budget
  • Disbursement conditions
  • Financing
  • Performance indicators and targets
  • Eligibility
  • Priority
  • Risk
Financial integration

Financial integration (FI) describes the components of GCPM that manage the relationship between GCPM and the financial system. This includes all coding required to set up accounts for direct deposit or automatic withdrawal for repayments, and ensures the structure of vendors and customer records are respected in the financial system without requiring multiple GCPM accounts to for separate bank accounts.

Portal - in brief

The portal is configured to display application forms designed in GCPM to portal users. Portal application forms can be

  • Static: a common set of fields and questions is displayed. In this case, each program / initiative may have a separate application form.
  • Interactive - as users enter information about their project and organization, the data presented to the user dynamically changes. Questions can be tailored based on input from the client

Portal accounts will interact with the GCPM account records. The portal will facilitate the creation of new accounts, and the management of client information, flagging where necessary when requested changes / updates are required.
